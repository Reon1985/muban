# 实施门禁检查

## 触发条件

在进入正式代码开发前自动触发，用于确认：

- 是否有已批准的实施文档
- 是否在批准范围内开发
- 是否已完成必要的前置阅读

## 核心原则

**没有已确认实施文档，不进入正式代码修改。**

### 2. 实施文档检查

必须确认：

- [ ] 当前任务是否有已批准的实施文档
- [ ] 实施文档是否覆盖本次改动范围
- [ ] 实施文档是否明确"不改内容"
- [ ] 实施文档是否明确风险点

**如果以上任一项为"否"，必须先完成实施文档审核，才能进入开发。**

### 3. 范围边界检查

必须确认：

- [ ] 本次改动目标是否清晰
- [ ] 本次改动范围是否明确
- [ ] 不改内容是否清楚
- [ ] 是否存在擅自扩功能风险

### 4. 阶段检查（模块型任务）

对于模块型任务，还需确认当前阶段：

- [ ] 当前处于"UI 确认阶段"还是"后端贯通阶段"
- [ ] UI 未确认前，不进入后端正式开发
- [ ] 如处于 UI 阶段，本次改动是否在已批准 UI 范围内

### 5. Agent 选择检查

必须确认：

- [ ] 是否已选定主开发 agent
- [ ] 是否已确定开发后审查 agent
- [ ] 选择依据是否与任务类型匹配

## 特殊情况

### 可以跳过门禁的情况

以下情况可以简化流程，不需要完整实施文档：

- 纯文案修正
- 局部样式修正
- 不影响结构的小 bug 修复
- 同一模块在已批准 UI 范围内的微调

但仍需确认：

- 改动范围极小
- 不影响其他模块或页面
- 不涉及菜单、路由、布局、业务规则

### 不能跳过门禁的情况

以下情况必须走完整审核：

- 改导航
- 改整体布局
- 改工作台结构
- 改模块边界
- 改重要业务规则
- 改多页面共用组件

## 注意事项

- 门禁检查不是"文档形式感"，而是保证改动有依据、边界有约束、风险有提示
- 即使是小改动，也要在心里过一遍范围边界
- 如果不确定是否需要门禁，默认先按需要处理
