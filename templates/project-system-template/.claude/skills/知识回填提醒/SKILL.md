# 知识回填提醒

## 触发条件

在以下情况触发：

- 探索到新的框架结论
- 发现新的技术限制、风险或坑点
- 形成新的可复用开发规范
- 任务阶段性完成但未做知识沉淀

## 核心原则

**探索得到的新框架认知，必须补回文档。**

未回填知识，不视为任务真正完成。

## 回填优先级

### A 级：必须立刻回填

满足以下任一情况时，必须在当前工作结束前立即回填：

- 当前实施文档的状态、范围、边界已经变化
- 当前模块需求、菜单结构、页面规则、业务口径已经变化
- 新发现会直接影响下一步开发判断的框架结论
- 新发现阻塞型风险、严重问题、关键限制

**一句话：不立刻回填，就可能导致下一步做错。**

### B 级：可按半天或一天批量回填

以下知识默认不要求每个小改动后立刻回填，可先记录"待回填清单"：

- 升级日志
- 效率提升
- 前端 / 后端说明的补充认知
- 框架说明类文档

**一句话：重要，但不直接阻塞当前开发。**

### C 级：可按阶段统一回填

以下知识默认在一个模块完成或阶段结束时再统一整理：

- README.md
- 项目文档总览.md
- 文档迁移、归档、导航类文档
- 阶段性总结类文档

**一句话：导航与归档类，不参与每轮微调。**

## 回填检查流程

### 任务完成前检查

每次任务收尾前，必须过一遍：

```markdown
## 知识回填检查

### 本次探索到的新知识
- [知识清单]

### 回填路径判断
- [知识 → 回填路径清单]

### 回填优先级判断
- A 级（立刻回填）：[清单]
- B 级（当日回填）：[清单]
- C 级（阶段回填）：[清单]

### 回填状态
- 已完成：[清单]
- 待完成：[清单]

### 最终结论
- [全部回填完成 / 有待回填项]
```

### 未回填时的处理

如果有待回填项：

- A 级未回填：**不视为任务完成，必须立刻回填**
- B 级未回填：记录到待回填清单，当日或下次集中回填时处理
- C 级未回填：记录到阶段待回填清单，阶段收尾时处理

## 注意事项

- 知识回填不是"额外工作"，而是项目资产积累
- 只停留在聊天里的知识，后续要么重新探索，要么遗忘
- 回填时使用规范化表达，不直接搬运聊天原话
- 回填后更新对应目录的 README.md 索引
