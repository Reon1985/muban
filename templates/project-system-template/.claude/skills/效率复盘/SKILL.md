# 效率复盘

## 触发条件

当本次正式开发累计时长超过 `8` 分钟时，收尾前必须执行效率复盘。

## 核心原则

效率复盘的目标：

- 回顾本次开发过程中的浪费
- 识别哪些问题应立即修正，哪些应沉淀为长期规则
- 让后续同类任务更快、更稳

**效率复盘是开发闭环的一部分，不是可选动作。**

## 复盘维度

必须覆盖以下六个维度：

### 1. 代码编写

- 是否有不必要的重复编写
- 是否有可复用但未复用的代码
- 是否有可模板化但未模板化的代码
- 是否有不必要的复杂度

### 2. 架构设计

- 是否有不必要的架构调整
- 是否有可预见的扩展但未预留
- 是否有不合理的拆分或耦合
- 是否有影响后续维护的设计决策

### 3. 文档编写

- 是否有不必要的文档重复
- 是否有可合并的文档
- 是否有缺失但影响后续的文档
- 是否有过度文档化

### 4. 项目流程方式

- 是否有不必要的流程等待
- 是否有可并行但串行的步骤
- 是否有可提前准备但未准备的依赖
- 是否有不必要的沟通轮次

### 5. Agent 协作方式

- Agent 选择是否合适
- Agent 调用次数是否合理
- Agent 之间是否有重叠工作
- Agent 结果是否有效利用

### 6. 验证与收尾方式

- 验证方式是否高效
- 是否有遗漏验证
- 收尾是否有遗漏项
- 回填是否及时完成

## 长期规则沉淀

### 判断是否沉淀为长期规则

如果复盘结论满足以下条件，应沉淀为长期规则：

- 适用于多个同类任务
- 可显著减少后续浪费
- 执行成本低或无成本
- 与现有规则不冲突

### 沉淀路径

| 规则类型 | 沉淀路径 |
|---------|---------|
| 项目级开发规则 | [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) |
| AI / rule / skill 执行机制优化 | [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md) |
| agent / skill 选型口径变化 | [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md) |

### 沉淀后更新

沉淀为长期规则后，必须：

1. 更新对应文档
2. 更新效率提升.md 的记录状态为"已执行"
3. 如影响 agent/skill，同步更新全局说明文档

## 注意事项

- 复盘不是"自我批评"，而是"资产积累"
- 复盘结论必须区分"本轮已修正"和"后续持续执行"
- 不要把临时问题沉淀为长期规则
- 不要因为复盘而过度增加流程负担
