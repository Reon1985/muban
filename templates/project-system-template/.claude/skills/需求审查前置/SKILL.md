# 需求审查前置

## 触发条件

当用户提出以下任一类型的修改意见时自动触发：

- 需求增删改
- 菜单结构调整
- 页面结构调整
- 交互方式调整
- 权限或数据范围口径调整
- 模块边界或实现范围调整

## 核心原则

**默认动作不是直接改页面或改代码，而是先审查需求合理性。**

## 执行流程

### 1. 判断修改类型

首先判断本次修改主要涉及：

- 业务规则 / 功能点范围
- 页面结构 / 信息组织
- 交互方式 / 体验流转
- 模块边界 / 结构拆分
- 实现范围 / 技术边界

### 2. 选择主审查 agent

根据修改类型选择 `1` 个最合适的主审查 agent：

| 修改类型 | 主审查 agent |
|---------|-------------|
| 业务需求、范围、功能点合理性 | `agency-product-manager` |
| 页面结构、交互方式、信息组织 | `agency-ui-designer` |
| 复杂交互与体验流转 | `agency-ux-architect` |
| 模块边界、通用能力复用、结构拆分 | `agency-software-architect` |

### 3. 判断是否需要补充 agent

如果修改同时跨两个以上维度，或存在明显边界、依赖、冲突风险，可追加 `1~2` 个补充审查 agent。

**agent 总数不超过 `3` 个**。

### 4. 执行审查

审查重点：

- 需求合理性
- 需求完整性
- 边界清晰性
- 潜在遗漏
- 与现有系统的冲突或依赖

### 5. 与用户完成聚焦讨论

- 先呈现审查结论
- 与用户确认理解是否正确
- 明确修改的最终范围和边界

### 6. 进入下一步

结论明确后，再进入：

- UI 修改（如处于 UI 阶段）
- 实施文档更新（如范围变化）
- 代码开发（如已进入开发阶段）

## 输出模板

```markdown
## 需求审查结论

### 本次修改类型
- [类型]

### 主审查视角
- [agent 名称]: [审查结论]

### 补充审查视角（如有）
- [agent 名称]: [补充风险/遗漏]

### 最终结论
- 合理性：[评估]
- 完整性：[评估]
- 边界：[明确边界]
- 风险点：[列出风险]
- 遗漏项：[列出可能遗漏]

### 建议下一步
- [建议动作]
```

## 注意事项

- 该规则是需求审查前置，不改变现有文档分级更新规则
- 高频 UI 微调也适用，但只需做轻量需求审视，不要求每次都产出独立正式文档
- 多 agent 参与时，补充 agent 只补充单一视角，不重复复述全案
- 最终结论由主线程统一合并并对外表达
