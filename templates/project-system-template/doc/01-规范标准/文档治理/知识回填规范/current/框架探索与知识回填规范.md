# 框架探索与知识回填规范

## 1. 文档定位

本文件用于定义本项目在“框架研究、源码探索、能力识别、知识沉淀”上的统一规则。

目标是把下面这套原则固化下来：

1. 每次工作优先查阅已有文档
2. 文档不足时再进入代码和框架探索
3. 探索到的新内容必须补回对应文档
4. 未完成知识回填，任务不视为真正完成

本文件属于项目级长期规则文件，适用于：

- AI
- 开发经理
- 项目经理
- 后续参与本项目的其他成员

---

## 2. 总原则

本项目默认采用以下知识工作链路：

```text
先查文档
→ 判断是否已有结论
→ 文档不足时再探索
→ 形成结论
→ 回填到对应文档
→ 再继续方案 / 实施 / 开发
```

一句话规则：

- 先用文档建立上下文
- 再用探索补齐缺口
- 再把探索结论回写为新文档资产

---

## 3. 强制规则

## 3.1 每次任务开始必须先查文档

任何任务开始时，默认先阅读：

1. [项目文档总览.md](__PROJECT_ROOT__/doc/项目文档总览.md)
2. [WORKFLOW.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/WORKFLOW.md)
3. [AGENTS.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/AGENTS.md)
4. 当前任务所属模块的最新正式文档

如果任务目标已经明确是“定位代码、接口、权限链或模块入口”，应在进入大范围代码探索前，优先补读：

1. [前端.md](__PROJECT_ROOT__/doc/00-入口/项目定位与环境/前端.md)
2. [后端.md](__PROJECT_ROOT__/doc/00-入口/项目定位与环境/后端.md)
3. [框架用户组织角色数据权限架构说明.md](__PROJECT_ROOT__/doc/01-规范标准/后端/权限与数据范围规范/current/框架用户组织角色数据权限架构说明.md)

只有在文档中找不到所需结论、或现有文档明显不完整时，才进入代码和框架探索。

## 3.2 探索必须带着明确问题进入

不允许“先随便看代码，再考虑要写什么文档”。

进入探索前，必须先明确：

- 要确认什么
- 为什么现有文档不够
- 本次探索希望补齐哪类结论

## 3.3 新发现必须回填文档

凡是本次探索得到的、对后续工作有复用价值的内容，都必须补回文档。

典型包括：

- 框架默认菜单结构
- 框架权限链路
- 用户、组织、角色、岗位之间的真实关系
- 数据权限实现方式
- 路由与布局机制
- 前后端接口约束
- 可复用组件能力
- 明显的框架限制和陷阱

## 3.4 未完成回填，不视为任务完成

如果本次工作中形成了新的框架认知，但没有补回文档，则该任务只能视为“部分完成”。

在这种情况下：

- 可以先给出阶段性结果
- 但在项目规范上，仍应继续完成知识回填

## 3.5 代码开发前必须先完成回填

如果探索内容会直接影响方案或代码实现，则必须：

1. 先回填文档
2. 再更新方案 / 实施文档
3. 再进入代码修改

不允许“探索出关键结论后只放在会话里，然后直接改代码”。

## 5. 探索后如何回填

## 5.1 按内容类型路由

### A. 项目通用、长期稳定、跨模块复用的框架结论

放到 `doc` 根目录的项目通用认知文档中。

例如：

- 框架用户组织角色数据权限架构说明
- 前端 / 后端总体说明
- 框架复用边界说明
- 布局与菜单机制说明

### B. 单次任务中新增的架构理解或模块方案

放到：

- `方案设计/架构设计/`
- `方案设计/技术方案/`

### C. 框架限制、踩坑、待确认项

放到：

- `问题总结/框架限制/`
- `问题总结/风险与待确认项/`
- `问题总结/踩坑记录/`

### D. 外部原始框架材料、截图、官方说明、参考图

放到：

- `参考文件/框架参考/`

### E. 本次变更导致的知识库更新记录

放到：

- `升级日志/`

## 5.2 回填时必须标明结论状态

为避免“猜测被当成事实”，所有框架结论建议标记以下状态之一：

- `已验证`
- `推断`
- `待验证`

推荐写法：

- `已验证：已阅读前端路由与权限代码，确认顶部一级菜单 + 左侧菜单 + Tab 页由框架现有 layout 能力提供。`
- `推断：当前菜单过滤依赖中文标题白名单，后续应替换为稳定标识。`
- `待验证：后端是否已有统一菜单摘要接口，当前尚未确认。`

## 6. 每类文档的补充要求

## 6.1 实施文档

每份实施文档建议增加或明确以下内容：

- 本次依赖的既有框架能力
- 本次新增探索结论
- 本次未完全确认的点
- 实施后需要沉淀回知识库的内容

## 6.2 日志记录

如果本次工作补充了重要框架认知，系统相关变更可记录到 `doc/05-升级日志/`，而 AI / 规则 / skill 自身优化应记录到 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)。

系统升级日志建议记录：

- 本次新增了什么认知
- 补到了哪份文档
- 对后续方案或开发有什么影响

补充规则：

- 规则优化、skill 优化、AI 执行约束优化，也视为一次正式升级
- 这类优化不能只改原则文件，必须同步补 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)
- 该日志中要明确写出：
  - 本次优化了什么规则或能力
  - 写入了哪些文档或 skill
  - 对后续执行一致性有什么帮助

## 6.3 问题总结

如果探索发现框架限制或潜在风险，不能只记在实施文档里，也要同步补到问题总结中。

## 7. 执行保障机制

为了尽量让这套规则“每次都被执行”，本项目采用以下多层保障：

1. 规则正式写入本文件
2. 规则同步写入 [WORKFLOW.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/WORKFLOW.md)
3. 规则同步写入 [AGENTS.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/AGENTS.md)
4. 规则同步写入 [doc目录结构与使用规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/文档结构规范/current/doc目录结构与使用规范.md)
5. 规则同步写入项目 skill：
   - [.codex/skills/project-doc-governance/SKILL.md](__PROJECT_ROOT__/.codex/skills/project-doc-governance/SKILL.md)

说明：

- 文档层负责“让规则可见、可审查、可协作”
- skill 层负责“让 AI 每次执行时优先命中这套流程”

## 7.1 需要明确的边界

这套机制可以显著提高执行一致性，但仍需诚实说明：

- 只要任务发生在本项目仓库上下文内，我可以按这套规则执行
- 如果未来换到其他仓库、其他线程、或没有加载这些项目文件的环境，需要重新把这套规则带过去

也就是说：

- 在本项目当前上下文中，这套规则可以作为默认执行规则
- 但它不是脱离项目环境后自动生效的全局规则

## 8. 任务完成前自检清单

在每次任务结束前，至少自检以下问题：

1. 我是否先查阅了现有文档？
2. 文档里已有结论是否已经被优先复用？
3. 本次是否进入了新增探索？
4. 如果有新增探索，是否形成了新的框架认知？
5. 这些认知是否已经回填到对应文档？
6. 是否需要补升级日志或问题总结？
7. 如果还没回填，是否应该明确说明任务尚未完全结束？

---

## 9. 当前正式结论

从本文件生效起，本项目默认采用：

- 文档优先
- 探索补缺
- 发现回填
- 回填完成后再继续实施

这条规则作为本项目当前正式知识治理规范。
