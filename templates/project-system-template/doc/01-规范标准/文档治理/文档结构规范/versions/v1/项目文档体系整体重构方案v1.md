# 项目文档体系整体重构方案 v1

## 1. 文档定位

本文件用于给当前项目提供一套**面向 AI 执行、面向版本管理、面向长期维护**的文档体系整体重构方案。

本方案不以“最小改动”为目标，而以以下三个目标为优先：

1. 降低 AI 每次启动任务时的默认阅读量
2. 让“统一规范”和“版本交付物”彻底分层
3. 让同一版本下的需求、前端、后端、数据库、测试、部署等内容形成完整版本包

本方案当前是**整体重构设计稿**，用于统一后续目录重建、文档迁移和入口改造，不等于本轮已经完成物理迁移。

---

## 2. 当前问题判断

当前仓库文档虽然已经完成了“根入口 + 一级索引 + 二级索引 + 正文”的第一轮治理，但随着版本推进和优化总结持续增加，已经出现以下结构性问题：

### 2.1 AI 阅读路径过长

同一个任务经常需要横跨多个一级目录：

1. 需求在 `需求/`
2. UI 相关内容在 `UI设计/`
3. 前端、后端、数据库、API 设计在 `方案设计/`
4. 具体测试与实施在 `实施文档/`
5. 历史原因、修复过程、优化结论又在 `升级日志/` 与 `问题总结/`

结果是：

1. AI 每次要先判断“这份信息到底归需求、UI、方案还是实施”
2. 同一个版本的内容分散在多个一级目录，版本上下文难以一次性收口
3. 目录入口虽然存在，但执行入口仍然不够短

### 2.2 统一规范与版本方案混读

当前 `UI设计/` 下同时存在：

1. 统一 UI 规范
2. 页面结构设计
3. 某个版本的页面升级方案

这导致一个常见误区：

1. 容易把“统一 UI 规范升级”误认为“某个业务版本自带一套独立 UI 规范”
2. 容易把标准层文档和交付层文档放在同一认知层级上

需要明确：

1. `UI规范` 是统一标准层，可以有标准版本升级，但不应被理解成“每个业务版本各自一套 UI 规范”
2. `版本内页面方案` 是交付层，回答的是“某个版本这批页面具体如何落地”

### 2.3 版本管理不完整

当前 `需求/` 已经具备版本目录，但前端、后端、数据库、API、测试、部署等版本内容仍然主要散落在不同一级目录下。

结果是：

1. `v1.1` 的需求、页面、数据库、API、测试文档无法自然聚合
2. 同一版本内的升级补丁缺少统一的版本级收口位
3. 后续 AI 做某个版本的工作时，仍要跨多个一级目录来回查找

### 2.4 优化总结越来越多，但执行入口不够固定

现在“优化总结、复盘、实施记录、升级日志、问题总结”都在持续增加。

如果继续沿用当前模式，AI 默认阅读量会持续上升，因为：

1. 历史证据和当前执行入口没有彻底分层
2. 标准操作手册尚未形成稳定主入口
3. 反思和优化结论没有被充分回写为“SOP”或“问题总结”

## 3. 重构目标

本次整体重构后的目标结构应同时满足以下要求：

### 3.1 对 AI 的目标

1. AI 默认只读最少数量的入口文档
2. AI 可以快速区分“统一标准”“当前版本交付物”“历史证据”
3. AI 做部署、发版、测试、页面检查等高频任务时，优先阅读标准操作手册，而不是长篇过程文档

### 3.2 对版本管理的目标

1. 每个业务版本形成完整版本包
2. 同一版本内的需求、前端、后端、数据库、测试、部署和升级记录可聚合查阅
3. 同一版本内的所有增量升级统一进入版本包下的 `升级/` 子目录

### 3.3 对长期规范的目标

1. UI 规范、前端实现规范、后端设计规范、数据库设计规范等统一进入标准层
2. 标准层可版本化升级，但保持统一性，不被业务版本切碎
3. AI 默认阅读标准层的 `current/`，只有在需要追溯旧版本规则时才看历史规范快照

### 3.4 对迁移治理的目标

1. 允许现有文档正文结构发生变化
2. 允许现有长文拆分、合并、改名、移动
3. 允许不再保留当前若干顶层目录的旧职责边界，只保留对新结构有意义的内容

## 4. 核心设计原则

### 4.1 统一标准层与版本交付层分离

后续所有正式文档首先要区分属于哪一层：

1. 标准层
   回答“长期应怎么做”
2. 操作层
   回答“现在执行这个高频动作时怎么做”
3. 版本层
   回答“某个版本交付了什么、怎么落地”
4. 证据层
   回答“这次为什么变、发生过什么问题、怎么复盘”

### 4.2 UI 规范保持统一性

后续必须明确：

1. `UI规范` 属于长期统一标准层
2. `UI规范` 可以按规范版本升级，例如 `v1 / v1.1 / v2`
3. `UI规范` 的版本号表示“标准演进版本”，不是“业务版本专项方案”
4. 某个业务版本中的页面设计稿、页面结构方案、页面升级清单，不再命名或归类为 `UI规范`

### 4.3 版本包承接具体交付，不承接统一规范

版本包内只承接：

1. 该版本的需求
2. 该版本的前端页面与前端落地方案
3. 该版本的后端方案
4. 该版本的数据库设计与脚本
5. 该版本的测试与部署文档
6. 该版本内的升级记录

版本包不承接：

1. 全项目统一 UI 规范
2. 全项目统一后端规范
3. 全项目统一数据库规范
4. 全项目统一测试规范

### 4.4 标准操作手册成为 AI 默认执行入口

部署、发版、测试、页面连通性检查、按钮点击与业务校验等高频动作，后续应优先通过 `标准操作手册` 承接。

流程应变成：

```text
优化 / 反思 / 实施结论
→ 回写标准操作手册
→ 回写问题总结
→ 过程证据再进升级日志
```

### 4.5 同一版本内升级统一收口

后续每个版本包内都必须带一个 `升级/` 子目录，用于放置该版本实施过程中的增量文档。

不再允许：

1. 同一版本的增量升级散落在多个一级目录
2. 某个版本的补丁同时挂在根目录、升级日志、实施文档、方案设计多个位置且无统一版本入口

## 5. 目标结构总览

本项目文档体系重构后，建议采用以下新顶层结构：

```text
doc/
├── 00-入口/
├── 01-规范标准/
├── 02-标准操作手册/
├── 03-版本包/
├── 04-问题总结/
├── 05-升级日志/
└── 99-参考与归档/
```

### 5.1 `00-入口`

负责提供最短阅读路径。

建议包含：

1. `README.md`
2. `AI执行入口.md`
3. `当前生效版本说明.md`
4. `目录导航与阅读顺序.md`

### 5.2 `01-规范标准`

负责统一长期标准，不承接业务版本具体交付物。

建议包含：

1. 产品规范
2. 前端规范
3. 后端规范
4. 数据库规范
5. 测试规范
6. 文档治理规范

### 5.3 `02-标准操作手册`

负责高频执行动作的 SOP。

建议包含：

1. 部署与发版
2. 测试执行
3. 页面连通性测试
4. 页面点击与业务校验测试
5. 回滚与应急处理

### 5.4 `03-版本包`

负责承接各个业务版本的正式交付物。

例如：

1. `v1.0/`
2. `v1.1/`
3. 后续 `v1.2/`、`v2.0/`

### 5.5 `04-问题总结`

负责承接“常见问题、定位方式、解决方案、规避建议”。

### 5.6 `05-升级日志`

负责保留系统、文档体系、标准操作手册、关键模块的变更证据。

### 5.7 `99-参考与归档`

负责沉淀：

1. 原始输入件
2. 历史参考
3. 已废弃文档
4. 临时探索过程参考

## 6. 规范标准层设计

### 6.1 目标结构

建议 `01-规范标准/` 采用以下结构：

```text
01-规范标准/
├── README.md
├── 产品/
│   ├── current/
│   └── versions/
├── 前端/
│   ├── UI规范/
│   │   ├── current/
│   │   └── versions/
│   ├── 页面实现规范/
│   ├── 组件复用规范/
│   └── 交互规范/
├── 后端/
│   ├── API设计规范/
│   ├── 服务设计规范/
│   └── 权限与数据范围规范/
├── 数据库/
│   ├── 表设计规范/
│   ├── 字段命名规范/
│   ├── 关系建模规范/
│   └── 脚本发布规范/
├── 测试/
│   ├── 测试设计规范/
│   ├── 测试报告规范/
│   └── 页面测试口径规范/
└── 文档治理/
    ├── 文档结构规范/
    ├── 命名与版本规则/
    └── AI阅读路径规范/
```

### 6.2 `current/` 与 `versions/` 的关系

标准层建议统一采用双层结构：

1. `current/`
   - 当前默认生效版本
   - AI 默认阅读入口
2. `versions/`
   - 历史规范快照
   - 只在兼容旧版本、追溯旧规则时再读取

### 6.3 UI 规范的正确归位

后续 `UI规范` 应这样管理：

1. 统一进入 `01-规范标准/前端/UI规范/`
2. 按“标准版本”演进，例如 `v1 / v1.1 / v2`
3. `current/` 始终指向当前生效的统一 UI 标准

需要强调：

1. `UI规范 v1.1` 表示“统一 UI 规范已升级到 v1.1”
2. 它不表示“业务版本 v1.1 自己有一套 UI 规范”
3. 某个版本的 PM 页面方案、页面清单、页面升级清单，应放在版本包的前端目录下，而不是放进 UI 规范目录

### 6.4 现有 UI 文档的未来边界

后续建议按下面边界重新组织：

#### 6.4.1 留在 UI 规范层的内容

1. 一期页面 UI 规范
2. 前端 UI 技术实现总纲
3. 前端组件抽取与复用原则
4. PM 新版视觉与交互中已经沉淀为全项目可复用的长期规则

#### 6.4.2 迁入版本包的内容

1. 某个业务版本的页面结构方案
2. 某个版本的页面重构方案
3. 某个版本的页面升级清单
4. 某个版本的新增页面设计与落地说明

## 7. 标准操作手册层设计

### 7.1 目标结构

建议 `02-标准操作手册/` 采用以下结构：

```text
02-标准操作手册/
├── README.md
├── 部署与发版/
├── 测试执行/
├── 页面连通性测试/
├── 页面点击与业务校验测试/
└── 回滚与应急处理/
```

### 7.2 每份 SOP 的固定结构

每份标准操作手册建议统一采用：

1. 适用范围
2. 前置条件
3. 标准步骤
4. 成功判定
5. 常见失败
6. 排查入口
7. 回滚方式
8. 关联问题总结
9. 关联版本包文档

### 7.3 与优化总结的关系

后续必须执行：

1. 每次高频动作有新优化结论，先更新对应 SOP
2. 每次出现可复现问题，更新问题总结
3. 本次过程证据再进入升级日志

不再默认新增“又一篇优化总结”作为 AI 的执行入口。

### 8.1 目标结构

每个业务版本建议采用统一模板：

```text

└── vX.Y/
    ├── README.md
    ├── 版本总览.md
    ├── 需求/
    ├── 前端/
    ├── 后端/
    ├── 数据库/
    ├── 测试/
    ├── 部署/
    └── 升级/
```

### 8.2 `需求/` 的目标结构

版本包中的需求目录建议只承接“做什么”的文档：

```text
需求/
├── 业务需求说明.md
├── 页面清单.md
├── 页面规则.md
├── 菜单与权限口径.md
└── 版本内需求升级/
```

后续不再把以下内容混入需求目录：

1. 后端实现细节
2. 数据库表设计
3. API 契约
4. 前端实现方案

### 8.3 `前端/` 的目标结构

版本包中的前端目录建议承接具体页面和落地方案：

```text
前端/
├── 页面结构方案/

├── 前端实现方案/
├── 路由与菜单承接方案/
└── 前端升级/
```

这里承接的是：

1. 该版本有哪些页面
2. 这些页面如何落地
3. 页面结构、导航承接、页面重构、风格试点如何执行

这里不再承接：

1. 全项目统一 UI 规范
2. 全项目统一前端实现总纲

后端/

## 10. AI 默认阅读路径

重构完成后，AI 默认阅读路径建议固定为：

```text
00-入口/AI执行入口.md
→ 00-入口/当前生效版本说明.md
→ 01-规范标准/对应领域/current
→ 02-标准操作手册/对应动作
→ 03-版本包/当前版本/README.md
→ 03-版本包/当前版本/目标文档
→ 04-问题总结/current（仅当异常时）
→ 05-升级日志 或 03-版本包/升级（仅当需要追溯）
```

### 10.1 AI 做需求任务时

```text
入口
→ 当前版本
→ 标准层中的产品/页面规则规范
→ 版本包/需求/
```

### 10.2 AI 做前端页面任务时

```text
入口
→ 当前版本
→ UI规范/current
→ 页面实现规范/current
→ 版本包/前端/
```

### 10.3 AI 做后端/API/数据库任务时

```text
入口
→ 当前版本
→ 后端规范/current 或 数据库规范/current
→ 版本包/后端/ 或 数据库/
```

### 10.4 AI 做部署、发版、测试时

```text
入口
→ 标准操作手册/对应动作
→ 当前版本/测试 或 部署
→ 如失败再看问题总结
```

## 11. 现有文档结构调整原则

本次整体重构不仅是目录迁移，也包含**正文结构重构**。

后续允许并建议对现有文档做以下调整：

### 11.1 允许拆分长文

当一份文档同时混有：

1. 统一标准
2. 业务版本内容
3. 实施过程说明
4. 升级补丁说明

则建议拆成多份正式文档。

### 11.2 允许合并同类碎片文档

若多篇文档本质上都在回答同一件事，例如：

1. 多篇页面测试补充说明

则允许合并为：

1. 一份标准操作手册

3. 一份版本级总文档 + 升级子文档

### 11.3 允许重命名

1. 文档层级
2. 主题
3. 版本
4. 生命周期

### 11.4 允许重写索引逻辑

1. 这个目录是什么层级
2. AI 应先看什么
3. 当前生效的入口是什么
4. 历史文档和当前文档如何区分

## 16. 当前结论

本次文档体系整体重构建议采用以下最终模型：

1. `00-入口`
2. `01-规范标准`
3. `02-标准操作手册`
4. `03-版本包`
5. `04-问题总结`
6. `05-升级日志`
7. `99-参考与归档`

其中最重要的边界是：

1. `UI规范` 是统一标准层，保持统一性，只做标准版本升级
2. 各业务版本只承接“版本内页面方案与交付物”，不再形成各自独立的 UI 规范体系
3. 高价值执行经验优先回写 `标准操作手册`
4. 同一版本的所有升级统一进入该版本包下的 `升级/`

本方案确认后，下一步应进入：

1. 新目录结构定稿
2. 现有文档迁移映射表编制
3. 第一批入口层、标准层、SOP 层文档骨架落地
