# doc目录结构与使用规范

## 1. 文档定位

本文件是当前项目 `doc/` 目录的正式结构规范。

它用于回答 4 个问题：

1. 现在 `doc/` 的正式结构是什么
2. 不同类型的文档应该放到哪一层
3. AI 和人工成员默认应该如何阅读
4. 新增、迁移、归档文档时应遵守什么规则

本文件描述的是**当前生效结构**，不是历史结构说明。

---

## 2. 当前正式结构

当前 `doc/` 正式采用以下 7 个主层级：

1. `00-入口`
2. `01-规范标准`
3. `02-标准操作手册`
4. `03-版本包`
5. `04-问题总结`
6. `05-升级日志`
7. `99-参考与归档`

对应含义如下：

| 层级 | 用途 |
| --- | --- |
| `00-入口/` | AI 默认启动入口、当前版本入口、阅读顺序、代码定位入口 |
| `01-规范标准/` | 长期稳定的统一规则，如 UI 规范、前后端规范、文档治理规范 |
| `02-标准操作手册/` | 高频执行动作的 SOP，如部署、发版、测试、回滚 |
| `03-版本包/` | 版本级正式交付物，如需求、前端、后端、数据库、测试、升级 |
| `04-问题总结/` | 当前问题、版本问题、故障排查、风险与治理清单 |
| `05-升级日志/` | 系统升级、版本升级、文档体系升级、AI 执行体系升级证据 |
| `99-参考与归档/` | 原始输入、历史稿、旧版参考、模板、页面参考、废弃文档 |

---

## 3. 阅读入口规则

### 3.1 默认阅读顺序

默认阅读顺序固定为：

```text
CLAUDE.md
→ doc/README.md
→ doc/00-入口/README.md
→ doc/00-入口/AI执行入口.md
→ doc/00-入口/当前生效版本说明.md
→ 目标标准 / 目标SOP / 目标版本包
```

### 3.2 开发任务额外阅读顺序

如果任务进入代码开发阶段，在上述基础上继续：

```text
00-入口/项目定位与环境/README.md
→ 前端.md / 后端.md / 权限架构说明
→ 01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md
→ 当前实施文档
→ 当前模块正式正文
→ 再进入代码
```

### 3.3 问题与追溯阅读顺序

如果遇到异常、历史差异或归档问题，默认继续：

```text
04-问题总结/
→ 05-升级日志/
→ 99-参考与归档/
```

## 4. 各层放置规则

### 4.1 `00-入口`

放置内容：

- 当前项目总入口
- AI 执行入口
- 当前生效版本说明
- 目录导航与阅读顺序
- 前端 / 后端 / 环境定位入口

不放：

- 长篇需求正文
- 具体版本实施文档
- 历史日志

### 4.2 `01-规范标准`

放置内容：

- 长期生效的统一规则
- 多版本共享的 UI 规范
- 前后端共性规范
- 文档治理、AI 阅读路径、知识回填规则

关键原则：

1. UI 规范属于统一标准层，不按每个业务版本各自拆一套标准。
2. 如有 UI 规范升级，应在 `UI规范/versions/` 下做版本升级，而不是放进某个业务版本包里。
3. 会长期影响多个任务判断的规则，优先进入这一层。

### 4.3 `02-标准操作手册`

放置内容：

- 部署与发版 SOP
- 页面连通性测试 SOP
- 页面点击与业务校验测试 SOP
- 回滚与应急处理 SOP

关键原则：

1. 高频重复动作优先 SOP 化。
2. SOP 写“怎么执行”，不写某个版本的具体业务方案。

### 4.4 `03-版本包`

放置内容：

- 某版本的需求
- 某版本的前端页面方案
- 某版本的后端技术方案
- 某版本的数据库设计
- 某版本的测试用例与测试报告
- 某版本的实施 / 升级文档

关键原则：

1. 同一版本内的需求、前端、后端、数据库、测试、升级统一收在同一版本包下。
2. 同一版本内的持续升级，统一放在该版本包内继续演进。
3. 版本包承接“这一版要怎么落地”，不承接长期统一规范。

### 4.5 `04-问题总结`

放置内容：

- 当前已知问题
- 故障排查
- 版本问题
- 风险与治理清单

关键原则：

1. 问题总结用于“下次再遇到同类问题时直接复用”。
2. 与版本强相关的问题可进入 `versions/<版本>/`。
3. 当前仍高频影响执行的问题优先放 `current/`。

### 4.6 `05-升级日志`

放置内容：

- 文档体系升级日志
- AI 执行体系升级日志
- 版本交付与联调日志
- 单次效率提升日志

关键原则：

1. 这里记录“发生了什么变化”和“证据”。
2. 正式规则本身仍要回写到 `01-规范标准/` 或 `02-标准操作手册/`。

### 4.7 `99-参考与归档`

放置内容：

- 原始输入
- 历史参考
- 旧版迁移稿
- 模板
- 页面参考图片
- 已废弃文档
- 历史占位文档

关键原则：

1. 能保留参考价值，但不再作为默认正式入口的材料，统一进入本层。
2. 历史日志仍需要引用的旧稿，如原件缺失，应在本层补稳定落点。

## 5. 当前推荐目录骨架

```text
doc/
├── README.md
├── 项目文档总览.md
├── 00-入口/
├── 01-规范标准/
├── 02-标准操作手册/
├── 03-版本包/
├── 04-问题总结/
├── 05-升级日志/
└── 99-参考与归档/
```

说明：

1. 当前 `doc/` 根目录只保留新结构目录，不再保留旧顶层兼容目录。
2. 历史目录内容应已迁移到新结构；如需追溯，使用迁移映射表和版本包/升级日志定位。
3. `项目认知/` 已不再作为兼容目录保留，能力已拆入 `00-入口/项目定位与环境/` 和 `01-规范标准/后端/...`。

## 6. 兼容目录规则

旧目录迁移规则：

1. 旧目录不再作为当前仓库目录结构的一部分维护。
2. 不再新建“兼容目录 README”作为过渡层。
3. 新的正式正文一律写入新结构目录。
4. 历史引用统一通过 `doc/README.md`、`doc/项目文档总览.md`、迁移映射表和版本包入口承接。

---

## 7. 新增文档时的更新规则

### 7.1 默认更新顺序

新增或迁移正式文档后，默认按以下顺序更新：

1. 先更新目标目录的 `README.md`
2. 如影响当前阅读入口，再更新 [doc/项目文档总览.md](__PROJECT_ROOT__/doc/项目文档总览.md)
3. 如影响根级阅读方式，再更新 [doc/README.md](__PROJECT_ROOT__/doc/README.md) 和 [CLAUDE.md](__PROJECT_ROOT__/CLAUDE.md)

### 7.2 必须更新索引的情况

满足以下任一情况时，必须同步更新对应索引：

1. 新增正式文档
2. 文档迁移或归档
3. 当前目录推荐阅读顺序变化
4. 当前目录新增子分类
5. 当前版本默认入口变化

## 8. 命名与版本规则

### 8.1 固定入口与治理文件

以下文件默认不在文件名中携带版本号：

- `README.md`
- `项目文档总览.md`
- `AI执行入口.md`
- `当前生效版本说明.md`
- `开发规范.md`
- `AI能力与规则升级日志.md`
- `doc目录结构与使用规范.md`

### 8.2 统一规范的版本升级

统一规范类文档采用：

```text
规范主题 + v版本号 + .md
```

例如：

- `一期页面UI规范v1.md`
- `PM新版视觉与交互规范v1.1.md`

### 8.3 版本包正文

版本包内正式正文可沿用业务含义明确的命名，不强制统一改名，但必须放在正确层级下。

### 8.4 单次日志

建议命名：

```text
主题 + 日志 + 日期.md
```

例如：

- `支付模块联调与发版回归日志2026-04-04.md`

## 9. 迁移与归档规则

### 9.1 迁移原则

迁移旧文档时，优先判断它属于哪一类：

1. 当前正式入口
2. 长期规范
3. 高频操作手册
4. 版本交付物
5. 问题总结
6. 升级证据
7. 历史归档

### 9.2 不再直接沿用旧分类

以下旧分类不再作为正式分类标准：

1. 按“需求 / UI / 方案 / 实施”四处分散的横切分类
2. 把统一规范和版本交付物混放
3. 把长期规则和单次日志混放

### 9.3 无原件历史引用处理

如果历史日志或旧实施文档仍引用某份旧稿，但原文已不存在：

1. 在 `99-参考与归档/` 下补稳定落点
2. 明确标注其为历史占位文档
3. 给出当前替代阅读入口

---

## 10. 一句话执行规则

1. 默认从 `00-入口/` 启动，不从旧目录散读开始。
2. 统一规则进 `01-规范标准/`，高频动作进 `02-标准操作手册/`。
3. 版本交付物统一进 `03-版本包/<版本>/`。
4. 问题、日志、归档分别进入 `04/05/99`，不要混写。
5. 兼容目录只保留跳转职责，不再承接新正文。
