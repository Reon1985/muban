# 问题修复文档归档与执行规范

## 1. 文档定位

本文件用于定义当前项目中“问题修复 / Bug 修复 / 缺陷收口”相关文档应该如何归档、如何起草、如何收尾。

它重点回答 5 个问题：

1. 修 Bug 是否默认需要单独开新版本
2. 当前版本修复、问题总结、升级日志、SOP 分别承接什么内容
3. 什么情况下应继续放在当前版本包中收口
4. 什么情况下应沉淀为 `04-问题总结/`
5. 什么情况下问题修复已经扩大为新的版本范围

相关入口：

1. [doc目录结构与使用规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/文档结构规范/current/doc目录结构与使用规范.md)

## 2. 核心结论

当前项目对“问题修复文档”的默认口径不是“逢修 Bug 必开新版本”，而是：

```text
先判断本轮修复属于哪一层
→ 再决定文档落点
→ 默认在当前承接版本内收口
→ 只有当修复已经扩大为新的正式需求范围时，才新开版本包
```

一句话规则：

- `单个 Bug 不默认单独开版本；同一版本内的问题修复，优先在当前版本包中继续收口。`

---

## 3. 默认判定顺序

遇到问题修复任务时，默认按下面顺序判断：

1. 这是不是当前生效版本或当前承接版本内的正式交付问题
2. 这是不是一条值得长期复用的排障知识或风险结论
3. 这是不是一次已经发生的修复、发版、验证事实，需要留证据
4. 这是不是一类会高频重复出现、需要 SOP 化或模板化的执行动作
5. 这轮修复是否已经扩大为新的业务范围、专项升级或新的版本主线

如果前 `4` 个问题已经能确定落点，就不默认新开版本。

## 4. 各层承接规则

### 4.1 `03-版本包/<版本>/`

默认放置：

1. 当前版本正式交付范围内的问题修复
2. 与本轮修复对应的实施文档、测试问题清单、测试报告、回归结论
3. 同一版本内的持续修复与持续收口

适用判断：

1. 问题直接影响当前版本需求、页面、接口、数据库、验证或发版
2. 修复完成后需要进入当前版本的实施、测试、联调或验收闭环
3. 同一批问题仍属于当前版本承接范围，没有形成新的独立需求包

强规则：

1. 同一版本内的持续修复，统一放在该版本包内继续演进
2. 不因为“只是修 Bug”就默认另开版本包
3. 如果本轮来源于测试报告、联调报告或 QA 缺陷清单，优先在当前版本包中冻结问题清单并集中收口

### 4.2 `04-问题总结/`

默认放置：

1. 可复用的问题分析
2. 故障排查结论
3. 风险与治理清单
4. 下次再遇到时可直接复用的踩坑经验

适用判断：

1. 这份内容主要回答“为什么会出问题”和“下次怎么排查”
2. 它的价值高于本次版本交付本身
3. 未来多个任务或多个版本都可能复用

补充规则：

1. 当前仍高频影响执行的问题，优先放 `current/`
2. 明显属于某个版本特有的问题，放 `versions/<版本>/`

### 4.3 `05-升级日志/`

默认放置：

1. 已经发生的修复事实
2. 发版、回归、验证、关闭结论
3. 与修复动作相关的证据和时间线

适用判断：

1. 当前主要需要回答“改了什么、何时改、怎么验证、结果如何”
2. 当前内容不是正式方案本身，而是事实留痕

强规则：

1. 升级日志只承接证据层，不承接长期规则正文
2. 如果同时存在实施文档、测试报告和日志，日志只写结果与证据，不重复承载完整规则

### 4.4 `02-标准操作手册/`

默认放置：

1. 会高频重复发生的问题修复执行流程
2. 通用检查卡片
3. 可直接套用的实施模板

适用判断：

1. 当前目标不是记录某次 Bug 本身，而是沉淀“以后这类 Bug 怎么做”
2. 需要把判断步骤、必填字段、发版与验证动作固化成模板或 SOP

## 5. 新开版本包的判断规则

只有满足以下任一情况时，才考虑新开版本包，而不是继续放在当前版本修复：

1. 本轮修复已经扩大为新的正式需求范围
2. 已不再是“当前版本收口”，而是新的专项升级包
3. 需要独立的需求、页面规则、数据库、API、实施和测试承接
4. 当前修复会形成下一阶段的正式版本主线

如果只是：

1. 修正当前版本内的实现缺陷
2. 补齐当前版本已确认需求中的漏实现
3. 对测试或联调问题做集中收口

则默认仍在当前承接版本内处理，不新开版本。

## 6. 实施模板选择规则

### 6.0 快速判断

| 场景 | 优先模板 | 关键判断 |
| --- | --- | --- |
| 来源于测试报告、联调报告或 QA 缺陷清单，且要统一发版验证 | [测试报告问题批量收口实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/测试报告问题批量收口实施模板.md) | 重点是冻结问题清单、批量收口、统一发版、统一验证 |
| 来源于单点缺陷、线上反馈、排障发现，但仍需正式实施文档 | [问题修复实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/问题修复实施模板.md) | 重点是归档判定、修复方案、验证计划与回填计划 |
| 来源于飞书 Bug 跟踪表，且本轮要按 `待修复 -> 待发版`、`待开发 -> 开发确认` 做批量判断和回写 | [飞书Bug修复与开发确认提示词模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/飞书Bug修复与开发确认提示词模板.md) | 重点是只查阅 `状态` 非空记录；对照当前需求判断；Bug 回写 `修复结果` 和 `测试结果`，新需求除回写 `修复结果` 和 `测试结果` 外，还同步回写 `状态 = 待开发确认` |
| 仍无法判断 | 先回到 [AI执行入口.md](__PROJECT_ROOT__/doc/00-入口/AI执行入口.md) | 先定任务类型，再选模板 |

### 6.0.1 固定字段最小集

无论使用哪一份问题修复模板，默认都应至少包含下面这组固定字段：

1. 任务名称
2. 当前承接版本
3. 问题来源
4. 是否测试报告驱动
5. 当前主归档层
6. 本轮范围 / 冻结范围
7. 修复方案或根因判断
8. 验证计划
9. 回填计划
10. 实施结论

补充规则：

1. 测试报告驱动模板可以把“本轮范围”展开为“本轮问题清单 + 冻结问题清单”
2. 通用问题修复模板可以把“修复方案”展开为“根因判断 + 修复思路 + 影响模块”
3. 可以按场景扩展字段，但不应删除这组最小字段

### 6.1 测试报告驱动修复

如果问题来源于测试报告、联调报告或 QA 缺陷清单，并且本轮需要统一发版和统一验证：

1. 优先使用 [测试报告问题批量收口实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/测试报告问题批量收口实施模板.md)
2. 同时遵循 [测试报告驱动问题批量收口规则.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/测试报告驱动问题批量收口规则.md)

### 6.2 非测试报告驱动的通用修复

如果问题来源于单点缺陷、线上反馈、排障发现或日常修复，但仍需要形成实施文档：

1. 优先使用 [问题修复实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/问题修复实施模板.md)
2. 在模板中显式写明：
   - 当前问题归档判定
   - 当前承接版本
   - 是否需要同步问题总结
   - 是否需要同步升级日志

### 6.3 飞书 Bug 跟踪表状态回写

如果问题来源于飞书 Bug 跟踪表，并且本轮核心目标是：

1. 找出 `测试结果 = 待修复` 且 `状态` 不为空的记录并推进到 `待发版`
2. 找出 `测试结果 = 待开发` 且 `状态` 不为空的记录并推进到 `开发确认`
3. 同步把判断结论写回 `修复结果`
4. 对新需求同步把 `状态` 回写为 `待开发确认`
5. 同步回填当前版本包、问题总结或升级日志

则优先使用 [飞书Bug修复与开发确认提示词模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/飞书Bug修复与开发确认提示词模板.md)。

## 7. 默认执行链

当前项目对问题修复默认采用以下收口链：

```text
问题提出
→ 判断归档层级
→ 选择合适模板
→ 在当前承接版本内修复或冻结问题清单
→ 本地验证 / 统一验证
→ 回填版本包
→ 回填升级日志
→ 如有长期复用价值，再补问题总结
```

---

## 8. 完成检查清单

本轮问题修复收口前，至少检查：

1. 是否已明确当前问题归属哪一层
2. 是否已明确当前承接版本
3. 是否误把单点 Bug 当成新版本包
4. 是否已补齐实施文档或测试问题清单
5. 是否已补齐升级日志或验证证据
6. 是否存在值得沉淀到 `04-问题总结/` 的复用经验

## 9. 完成标准

一轮问题修复只有在下面条件满足后，才视为完成：

1. 已明确当前问题归属哪一层。
2. 已明确当前承接版本。
3. 已选对实施模板或执行路径。
4. 已完成必要验证与事实回填。
5. 已明确是否还有需要继续补充的问题总结或规则沉淀。
