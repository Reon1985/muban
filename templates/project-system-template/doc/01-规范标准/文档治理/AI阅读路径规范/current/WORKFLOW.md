# WORKFLOW.md

## 1. 文档定位

本文件定义“所有项目通用”的标准工作流。

目标是让所有项目在推进时，都遵循统一的节奏，而不是每次临时决定流程。

---

## 2. 总体工作流

建议所有项目默认采用以下主流程：

```text
问题提出
→ 需求梳理
→ 范围确认
→ 方案设计
→ 实施文档
→ 审核确认
→ 开发实施
→ 验证与修正
→ 效率复盘与改进
→ 升级日志
→ 问题总结 / 复盘
```

补充原则：

- 在进入上述主流程前，默认先查阅已有文档建立上下文
- 如果文档不足，再进入框架或代码探索
- 探索到的新框架内容，必须补回对应文档后，才视为真正完成本次工作
- 需求、页面、规则、方案、数据库、接口、实施文档、测试结论和代码实现等关键任务，默认采用“自动选择主 agent 执行 -> 自动选择独立复核 agent 审查 -> 主线程统一收口”的三段式协作
- 对于模块型工作，默认采用“先确认需求 / 菜单 / 页面规则，先做 UI 预览并确认，再进入后端实施”的双阶段节奏
- 从 `v1.1` 起，新版本或大版本增量默认采用“先系统需求说明书，再页面清单 / 规则清单 / UI 预览，再数据库架构 / API 清单”的标准流程
- 如果新版本需求会改动原有已联调页面，确认前不直接修改原页面，先新增本地预览页面或预览路由承接讨论

关联规则详见：

- [框架探索与知识回填规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/知识回填规范/current/框架探索与知识回填规范.md)

## 3. 各阶段说明

## 3.0 前置动作：文档优先

在任何正式任务开始前，默认先做以下动作：

1. 查阅 [CLAUDE.md](__PROJECT_ROOT__/CLAUDE.md)
2. 查阅 [项目文档总览.md](__PROJECT_ROOT__/doc/项目文档总览.md)
3. 进入当前任务对应目录的 `README.md`
4. 查阅当前模块相关正式正文
5. 如果本次任务会进入代码开发，再查阅 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) 和当前实施文档
6. 如果本次任务涉及开发 agent 选型、审查 agent 选型，或会调整 agent / skill 规则，再按需查阅 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)

前置协作强制规则：

1. 只要属于关键任务，开始前默认先自动选择 `1` 个最合适的主 agent。
2. 关键任务主结果形成后，默认再自动选择 `1` 个最合适的独立复核 agent。
3. 只有轻量无语义改动，才允许停留在主线程自检。

如果文档中已有结论，应优先复用，不重复探索。

只有以下情况才进入探索：

- 文档中没有该结论
- 文档明显过期
- 本次问题必须通过源码或运行结果验证

强约束：

- 探索得到的新框架认知，必须补回文档
- 未回填知识，不视为任务真正完成

## 3.1 问题提出

输入可能包括：

- 新需求
- 页面改版
- 规则调整
- 框架研究
- 问题修复

本阶段目标：

- 明确这次是在解决什么问题
- 判断任务属于需求、设计、实施还是修复

输出：

- 初步任务定义

## 3.2 需求梳理

本阶段目标：

- 统一目标
- 统一范围
- 统一术语

需要回答：

- 做什么
- 不做什么
- 一期做到什么程度
- 哪些属于后续扩展

输出：

- 需求说明书
- 页面清单
- 规则清单

### 3.2.1 新版本标准实施流程

从当前项目 `v1.1` 开始，所有“新版本增量”默认优先采用以下顺序：

```text
系统需求说明书
→ 页面清单
→ 页面规则清单
→ UI 界面 / 本地预览页
→ 页面确认
→ 数据库架构设计
→ API 接口清单
→ 实施文档
→ 审核确认
→ 开发实施
```

执行规则：

1. 需求阶段只定义业务目标、页面范围、字段口径、角色权限和关键规则，不提前混入数据库字段实现与接口细节。
2. 页面清单、规则清单和 UI 预览确认前，不进入正式数据库设计和 API 设计。
3. 如果需求涉及原有正式页面改造，确认前必须先新增本地预览页面或预览路由，不直接改原页面。
4. 本地预览页应以目标正式页面为承接对象：涉及既有正式页面时，默认保留正式页的搜索项、核心字段和操作语义，并在已确认新需求范围内承接结构或交互升级，不做脱离正式页回切目标的全新原型。
5. 本地预览页默认直接按目标正式页面形态组织，不额外堆放规则说明型提示块、页面说明副标题或评审导向的大段解释文案；如与正式页的确认弹窗、禁用条件、校验范围或触发入口暂未完全对齐，必须显式标注差异。
6. 本地预览页要与原页面、原接口调用和原联调链路隔离，避免影响已跑通页面。
7. 只有在页面与交互确认后，才进入数据库架构设计、API 清单和后续实施文档阶段。
8. 页面确认后，再决定是替换原页面、合并回原页面，还是继续保留独立正式页面。

## 3.3 范围确认

本阶段目标：

- 锁定本次边界
- 避免边做边扩

需要确认：

- 本次只做文档，还是做代码
- 本次只做原型，还是做真实功能
- 本次只改 UI，还是改业务逻辑

输出：

- 范围结论

补充规则：

- 模块型任务应进一步明确：本轮处于“UI 确认阶段”还是“后端贯通阶段”
- 默认先完成 UI 确认阶段，再进入后端贯通阶段
- 如果本轮属于新版本需求探索，且会影响既有正式页面，范围确认时必须额外说明“当前是否仅做本地预览页，不改原页面”

## 3.4 方案设计

本阶段目标：

- 明确怎么做
- 明确复用与新增边界
- 明确技术与页面结构
- 明确本次依赖了哪些既有框架能力
- 明确本次新增探索结论有哪些

适合产出：

- 架构方案
- UI方案
- 模块拆分方案
- 路由与导航方案
- 本地预览页方案

输出：

- `方案设计` 目录中的正式方案文档

## 3.5 实施文档

本阶段目标：

- 将“准备怎么改”整理成审核材料

适用场景：

- 重要结构改动
- 跨页面改动
- 跨模块改动
- UI 改版
- 影响较大的功能调整

实施文档至少应包含：

- 本次改动目标
- 本次改动范围
- 不改内容
- 影响代码概述
- 风险点
- 推荐实施顺序
- 本次依赖的既有框架能力
- 本次新增探索结论
- 本次实施后需补录的知识沉淀项

输出：

- `实施文档/待审核`

补充规则：

- 同一模块的 UI 高频微调，可在已批准实施范围内连续推进
- 只有当菜单结构、页面规则、交互流程或后端实现范围发生实质变化时，才需要更新实施文档并重新审核

## 3.6 审核确认

本阶段目标：

- 让开发经理、项目经理确认是否可以进入实施

审核重点：

- 方向是否正确
- 范围是否清晰
- 影响是否可控
- 风险是否已提示

输出：

- 审核结论
- 必要的补充修改

强约束：

- 如果本次任务涉及代码改动，未完成实施文档审核确认前，不允许进入代码修改阶段。

## 3.7 开发实施

本阶段目标：

- 按已审核的范围落地

要求：

- 不擅自扩大边界
- 不跳过关键结构改动说明
- 改动过程中同步维护必要文档
- 未经确认的实施方案，不允许直接修改代码

开发阶段的完整执行规则（agent 选型、审查机制、文档回填、收尾清单等）以 [开发规范.md](开发规范.md) 为准。

输出：

- 代码改动
- 页面改动
- 必要文档更新

## 3.8 验证与修正

本阶段目标：

- 确认实现结果与目标一致

验证内容：

- 页面结构
- 导航逻辑
- 业务交互
- 范围一致性
- 代码审查结论

输出：

- 验证结果
- 修正项

## 3.9 效率复盘与改进

本阶段目标：

- 回顾本次开发过程中的浪费
- 识别哪些问题应立即修正，哪些应沉淀为长期规则
- 让后续同类任务更快、更稳

触发规则：

- 只要本次任务进入正式开发且累计时长超过 `8` 分钟，收尾前必须执行本阶段

最低复盘维度：

- 代码编写
- 架构设计
- 文档编写
- 项目流程方式
- agent 协作方式
- 验证与收尾方式

输出：

- 当次效率提升记录
- 必要的治理文档回填
- 必要的规则优化项

补充要求：

- 复盘时要明确区分“真正的关键路径阻塞”与“平台或工具层噪音”，优先把会反复阻塞交付的动作前置成预检清单
- 如果本轮耗时主要来自 agent 平台故障、长时间等待或串行重试，必须把“主线程本地兜底时机”和“替代复核时机”写回正式规则
- 如果本轮出现部署脚本、升级 SQL、权限菜单、静态快照等高风险点被过晚发现，后续同类任务默认把这些检查前置到独立复核之前

## 3.10 升级日志

本阶段目标：

- 记录这次改了什么

建议记录：

- 变更主题
- 变更时间
- 影响范围
- 版本号
- 本次是否补充了新的框架或底座认知
- 新认知被补到了哪些正式文档
- 本次是否调整了 AI 执行规则、skill 约束或知识治理机制

补充要求：

- 系统、需求、UI、模块本身的升级，记录到 `升级日志`
- 规则优化、skill 优化、AI 工作流优化，记录到 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)

输出：

- 系统升级：`升级日志`
- AI / 规则 / skill 升级：[AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)

## 3.11 问题总结 / 复盘

本阶段目标：

- 沉淀经验
- 沉淀问题
- 为后续迭代减少重复沟通和重复踩坑

适合记录：

- 已知问题
- 方案取舍
- 框架限制
- 踩坑总结
- 探索中发现但暂未解决的框架缺口

输出：

- `问题总结`

## 4. 哪些任务可以简化流程

不是所有任务都要走完整链路。

以下任务可以简化：

### 4.1 小修类任务

例如：

- 文案调整
- 单页样式优化
- 小范围页面修复

可采用：

```text
问题提出 → 实施 → 审查 → 验证 → 效率复盘与改进（如超过 8 分钟）→ 升级日志
```

### 4.2 中等改动

例如：

- 单模块页面结构调整
- 单功能规则优化

可采用：

```text
问题提出 → 需求梳理 → 实施文档 → 审核 → 实施 → 审查 → 验证 → 效率复盘与改进（如超过 8 分钟）
```

### 4.3 大改动

例如：

- 导航重构
- UI 改版
- 跨模块重构
- 重要业务规则重构

建议走完整流程：

```text
问题提出 → 需求梳理 → 范围确认 → 方案设计 → 实施文档 → 审核 → 实施 → 验证 → 效率复盘与改进 → 升级日志 → 问题总结
```

## 5. AI 参与时的标准流程

AI 参与项目时，默认遵守以下顺序：

1. 先走双层索引查文档
2. 再判断是需求、UI、方案、实施还是开发任务
3. 重要改动先产出实施文档，再交审核
4. 模块型任务先做 UI 预览与确认，再进入后端
5. 开发完成后继续做验证、复盘、日志和知识回填

具体开发细则以 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) 为准。

## 6. 文档与目录配合规则

各阶段输出目录继续保持如下映射：

| 阶段 | 输出目录 |
|---|---|
| 需求梳理 | `需求` |
| 方案设计 | `方案设计` |
| UI 设计 | `UI设计` |
| 实施审核 | `实施文档` |
| 升级记录 | `升级日志` |
| 问题沉淀 | `问题总结` |
| 外部输入原件 | `参考文件` |

目录内具体先看哪份，不在本文件平铺维护，统一进入对应目录的 `README.md`。

## 7. 审核门槛建议

以下情况，建议不得跳过审核：

1. 改导航
2. 改整体布局
3. 改工作台结构
4. 改模块边界
5. 改重要业务规则
6. 改多页面共用组件

以下情况可不单独走审核文档：

1. 纯文案修正
2. 局部样式修正
3. 不影响结构的小 bug 修复

---

## 8. 一句话流程原则

所有项目统一采用：

“先定义问题和边界，再设计与审核，最后实施、验证、复盘与留痕；重要改动先出文档，避免无方案直接开发。”
