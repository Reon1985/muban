# AGENTS.md

## 1. 文档定位

本文件定义“所有项目通用”的 AI / 角色协作方式。

目标不是列出某个平台全部 agent 名单，而是规定：

- 项目中需要哪些角色视角
- 不同任务应由谁主导
- 何时需要评审
- 各类输出物由谁负责

说明：

- 本文件负责“角色模型、主导关系、协作边界”
- 具体开发阶段的 agent 选型、审查细则和执行门禁，以 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) 为准
- 具体 agent / skill 名称、推荐搭配和默认选型参考，以 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md) 为准

## 2. 项目中的标准角色

建议将项目中的 AI / 人工协作角色，统一抽象为以下几类。

### 2.1 需求与业务角色

职责：

- 梳理需求
- 确认范围
- 固化页面清单
- 固化业务规则

适用任务：

- 需求说明书
- 页面清单
- 字段清单
- 指标口径
- 规则梳理

### 2.2 架构与技术角色

职责：

- 梳理系统结构
- 判断复用与扩展边界
- 评估改动影响
- 输出技术方案

适用任务：

- 架构研究
- 框架研究
- 技术方案
- 模块拆分
- 重构影响分析

### 2.3 UI / 交互角色

职责：

- 梳理页面结构
- 统一导航方式
- 统一视觉和交互规范
- 评估页面改版方案

适用任务：

- UI规范
- 页面结构设计
- 导航设计
- 交互改版

### 2.4 实施审核角色

职责：

- 审核本次改动是否合理
- 审核影响范围是否清楚
- 审核实施顺序与风险

适用任务：

- 单次改动实施文档
- 开发经理审核材料
- 项目经理审核材料

### 2.5 开发执行角色

职责：

- 按已确认方案落地代码
- 保证边界不漂移
- 同步文档与页面实现

前提：

- 必须基于已确认的实施方案执行
- 如果实施方案未确认，开发执行角色不得直接改代码

适用任务：

- 页面开发
- 组件开发
- 路由调整
- 布局改造
- 功能实现

### 2.6 质量与复盘角色

职责：

- 做自检
- 做 QA
- 做问题归档
- 做复盘总结

适用任务：

- 验证页面
- 问题总结
- 升级日志
- 复盘文档

## 3. 标准协作规则

### 3.0 AI 的默认前置动作

AI 在每次正式任务开始前，默认按 [WORKFLOW.md 3.0](WORKFLOW.md) 中定义的文档优先规则执行前置阅读。

核心原则：

- 文档已有结论，优先复用
- 文档不足，再进入源码和框架探索
- 探索出新的框架认知后，必须补回正式文档
- 如果没有完成知识回填，则任务只算阶段性完成，不算正式收口

### 3.1 一次任务只允许一个主导角色

一个任务可以有多个视角参与，但必须明确一个主导角色。

例如：

- 写需求：需求与业务角色主导
- 写架构方案：架构与技术角色主导
- 做 UI 改版：UI / 交互角色主导
- 写实施文档：实施审核角色主导

### 3.2 评审角色不能替代主导角色

评审的职责是：

- 找问题
- 找风险
- 找遗漏

不是重新定义任务本身。

### 3.3 先产出主结果，再做独立复核

推荐顺序：

1. 先由 `1` 个主 agent 产出主结果
2. 再由 `1` 个独立复核 agent 做问题检查与风险审视
3. 再由主线程统一整合、定稿或决定是否返修

适用范围：

- 需求说明书
- 页面清单与规则清单
- UI 方案与页面结构设计
- 数据库架构设计
- API 接口清单
- 实施文档
- 代码实现

复核重点：

- 是否遗漏功能点、字段口径、页面规则或边界条件
- 是否存在规则冲突、表达歧义或与既有正式文档不一致
- 是否具备完整性、全面性与可实施性
- 是否符合项目文档规范、UI 规范、架构规范和开发规范
- 是否存在后续高风险返工点

复核结果至少应明确回答：

- 是否发现遗漏
- 是否发现冲突或不一致
- 当前结果是否可实施 / 可继续推进
- 是否允许进入下一阶段

不建议多个角色同时各写一版正式稿，也不建议主 agent 自己兼任默认复核角色。

### 3.4 框架探索不能只停留在会话里

AI 或其他角色在任务过程中探索出的新框架内容，不能只停留在聊天记录里。

必须按内容类型补回：

- 项目通用认知文档
- 方案设计文档
- 问题总结文档
- 升级日志

适用内容包括：

- 框架菜单结构
- 布局机制
- 用户组织岗位角色关系
- 数据权限实现方式
- 审批流和待办机制
- 文件、日志、接口等系统能力
- 任何会影响后续方案或开发的框架事实

### 3.5 开发阶段的 agent 协作原则

凡是进入正式开发阶段的任务，默认执行以下原则：

1. 自动选择 `1` 个主开发 agent
2. 如需补充视角，可追加辅助 agent，但不替代主开发 agent
3. 开发完成后，自动选择独立审查 agent 做代码审核
4. 审查重点优先看正确性、范围漂移、回归风险和缺失文档回填
5. 主线程不再把“自己直接写”作为默认起点，而是先明确由哪个开发 agent 主做

说明：

- 本文件只定义“开发阶段必须有主开发 agent 和独立审查 agent”这一协作边界
- 具体如何选、优先选谁、什么时候追加补充审查，统一以 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) 为准

### 3.6 全工作 agent 协作原则

凡是处于项目工作流中的任何任务，默认都必须先调取合适的主 agent 处理。

覆盖范围：

- 写文档
- 数据库设计
- 接口设计
- 页面 UI 设计
- 测试与验证
- 部署、发版、联调、排障
- 复盘、治理和交接整理

默认执行原则：

1. 每项工作先选 `1` 个最合适的主 agent
2. 主结果形成后，默认再选 `1` 个独立复核 agent 做审查
3. 如任务明显跨多个维度，可按需追加补充 agent，但仍应保持“主 agent + 独立复核 agent”为最小稳定骨架
4. 主线程负责统一收口、整合结论和对外沟通
5. 不因可并行就无差别堆叠多个 agent，仍以“最合适、最少必要”为原则
6. 只有在“没有合适 agent / agent 平台不可用 / 用户明确禁止使用 agent”时，才允许主线程直接接管

说明：

- 本文件定义的是“所有工作默认必须先找合适主 agent，并在关键产出后做独立复核”这一协作边界
- 对于轻量无语义改动，可由执行细则定义例外，不必把所有微小修订都升级为完整双 agent 流程
- 具体选型参考和默认推荐组合，统一以 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md) 和 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) 为准

### 3.7 新开发规范必须回填文档

如果开发过程中形成了新的可复用开发规范，必须写回：

- [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md)

如果同时影响 AI 执行规则、skill 路由或治理机制，还必须同步写回：

- [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)

如果同时影响 agent / skill 的用途说明、推荐搭配或默认选型规则，还必须同步写回：

- [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)

## 4. 各类任务推荐主导角色

## 4.1 写需求类文档

主导角色：

- 需求与业务角色

协同角色：

- 架构与技术角色
- UI / 交互角色

输出：

- 需求说明书
- 页面清单
- 业务规则

## 4.2 写架构研究与技术方案

主导角色：

- 架构与技术角色

协同角色：

- 需求与业务角色
- 实施审核角色

输出：

- 架构说明
- 技术方案
- 框架研究

## 4.3 写 UI 规范与页面结构方案

主导角色：

- UI / 交互角色

协同角色：

- 需求与业务角色
- 架构与技术角色

输出：

- UI 规范
- 导航方案
- 页面结构设计

## 4.4 写实施文档

主导角色：

- 实施审核角色

协同角色：

- 架构与技术角色
- 开发执行角色

输出：

- 单次改动实施文档
- 审核包

说明：

- 这类文档默认面向开发经理和项目经理
- 重点不是讲理念，而是讲本次改动

## 4.5 正式写代码

主导角色：

- 开发执行角色

协同角色：

- 架构与技术角色
- UI / 交互角色
- 质量与复盘角色
- 代码审查角色

输出：

- 页面
- 组件
- 路由
- 样式
- 功能实现
- 开发后审查结论

## 4.6 做复盘与问题总结

主导角色：

- 质量与复盘角色

协同角色：

- 开发执行角色
- 实施审核角色

输出：

- 问题总结
- 踩坑记录
- 升级日志

## 6. 默认协作动作

### 6.1 收到需求或方案类任务

默认动作：

1. 先判断任务类型
2. 先确定主导角色
3. 再决定是否需要补充评审视角

### 6.2 收到开发类任务

默认动作：

1. 先判断是否已经进入开发门禁范围
2. 若需要实施审核，先补实施文档
3. 确认后再进入开发执行

说明：

- 哪些任务必须出实施文档、哪些可以简化流程，以 [WORKFLOW.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/WORKFLOW.md) 和 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md) 为准
- 本文件只负责说明“谁主导、谁协同、谁评审”

---

## 7. 一句话原则

所有项目统一采用：

“一个任务一个主导角色，多个角色做互补评审；正式开发前，先明确主文档、边界和审核责任。”
