# 文档与 skill 去重治理规范

## 1. 文档定位

本文件用于定义文档体系和 skill 体系持续增长后，如何做去重治理和权威源校准。

本文件回答的问题是：

1. 什么才叫“重复”
2. 去重的目标是什么
3. 文档和 skill 应如何判断保留、合并、降级、归档
4. 去重后哪些入口必须同步更新
5. 哪些内容应该写进正式文档，哪些应该写进 skill

本文件是长期治理规则。

## 2. 去重治理的目标

去重治理的目标不是“数量最少”，而是：

1. 权威源清晰
2. 阅读入口清晰
3. 规则边界清晰
4. 冲突最小
5. 长期维护成本可控

一句话：

- 治理的目标是降低认知冲突，不是机械删文档。

---

## 2.1 权威源分层

当前项目默认按下面三层理解权威关系：

1. 正式文档
   - 负责长期规则、流程顺序、审核门禁、文档落点、命名规则、模板使用原则
   - 是默认权威源
2. 项目专属 skills
   - 负责把正式规则转成当前仓库可执行的入口链、任务分型、最小检查清单和调用提示
   - 是执行 adapter，不是新的长期规则源
3. 全局 core / 角色型 skills
   - 负责跨项目可复用的方法、角色视角和流程能力
   - 不能覆盖当前项目正式文档的结论

出现冲突时，默认优先级：

`正式文档 > 项目专属 skill > 全局 skill`

一句话：

- 文档定规则，项目 skill 做适配，全局 skill 提供通用能力。

---

## 3. 什么算重复

以下情况视为重复或高冲突候选：

1. 两份文档回答的是同一个问题
2. 两个 skill 的默认职责高度重叠
3. 多个入口同时声称自己是权威源
4. 新文档只是旧文档的改写副本，没有新增边界
5. README 和正文都在重复承载完整规则

---

## 4. 文档治理动作

发现重复文档时，默认只允许以下治理动作：

1. 保留一个权威正文
2. 其他位置改为索引或引用
3. 合并内容后归档旧稿
4. 如仍有历史价值，移入 `99-参考与归档/`

默认不建议：

1. 在多个位置长期维护同一条完整规则
2. 因为担心丢信息而无限保留平行正文

---

## 5. skill 治理动作

发现 skill 重叠时，默认按以下顺序处理：

1. 先判断职责是否真的不同
2. 如果只是入口不同但职责相同，应合并职责描述
3. 如果一个是全局 core，一个是项目 adapter，应明确边界，不算重复
4. 如果某 skill 已失效，应在说明文档中下线并清理入口

一句话：

- “分层不同”不等于重复，“职责相同却各写一份”才是重复。

---

## 5.1 文档与 skill 承载边界

以下内容默认写入正式文档，不应只写在 skill 中：

1. 长期生效的项目原则
2. 标准工作流与阶段顺序
3. 文档分层、命名和路由规则
4. 审核门禁、批准标准和例外口径
5. Agent / Skill 默认选型规则和协作边界
6. 模板和 SOP 的使用原则

以下内容默认写入 skill，不必在正式文档中展开为大段执行细节：

1. 某类任务触发该 skill 的条件
2. 最小必读文档清单
3. 当前仓库的入口链和引用顺序
4. 任务分型后的下一步动作
5. 该 skill 需要加载哪些 references

以下内容需要文档和 skill 同时出现，但职责不同：

1. 项目专属规则已经进入正式文档，skill 只做精简执行适配
2. 项目专属 skill 的用途、边界和推荐搭配，要在正式文档留总入口
3. skill 发生调整时，要同步更新对应权威文档，不允许只改 skill 不改文档

一句话：

- 能影响长期判断的内容写进文档，能减少执行摩擦的内容写进 skill。

---

## 6. 去重判断标准

每次治理时，至少检查以下问题：

1. 这份文档或 skill 解决的问题是否唯一
2. 是否已有更高层级的权威源
3. 是否仍被真实流程引用
4. 删除或合并后，入口是否还能命中
5. 是否需要历史追溯

如果不能回答上述问题，不应直接删除。

---

## 6.1 变更落点判断

遇到 Agent / Skill / 文档治理相关变更时，默认先判断落点：

1. 如果变更的是长期规则、默认口径或目录结构，先改正式文档
2. 如果变更的是项目仓库的执行适配、阅读路径或调用提示，再改项目专属 skill
3. 如果变更的是全局可复用方法论，优先改全局 core skill
4. 如果变更同时影响正式规则和执行适配，文档与 skill 必须同轮更新

不建议：

1. 先改 skill，等以后再补文档
2. 在多个 skill 内分别维护同一条项目规则
3. 用长篇 skill 正文代替正式文档

---

## 7. 必须同步更新的入口

只要做了文档或 skill 去重治理，默认至少检查：

1. 目标目录 `README.md`
2. [doc/README.md](__PROJECT_ROOT__/doc/README.md)
3. [项目文档总览.md](__PROJECT_ROOT__/doc/项目文档总览.md)
4. [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)
5. 如属规则升级，再更新 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)
6. 如属项目专属 skill 升级，检查仓库 `.codex/skills/` 对应 `SKILL.md`
7. 如属项目专属 skill 升级，检查对应 `agents/openai.yaml` 是否仍与 `SKILL.md` 一致

---

## 8. 推荐治理节奏

建议按以下节奏治理：

1. 每个版本收口时做一次轻量盘点
2. 每次新增长期规则或新增 skill 后做一次局部去重检查
3. 每次迁移到新项目后做一次跨项目去重检查
4. 每次项目专属 skill 调整入口链、默认提示或职责边界后，做一次文档与 skill 对齐检查

---

## 9. 完成标准

文档 / skill 去重治理机制建立完成，至少满足：

1. 已定义重复判断标准
2. 已定义文档治理动作
3. 已定义 skill 治理动作
4. 已定义必须同步更新的入口
5. 已定义推荐治理节奏
6. 已定义文档与 skill 的承载边界
