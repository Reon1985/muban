## 2. 一句话概括

当前项目采用的是一套“文档优先、实施有门禁、知识必须回填、AI 与人工协同推进”的运作体系。

默认节奏是：

```text
先查文档
→ 先做需求审查
→ 先确认页面与 UI
→ 再做实施文档
→ 审核通过后再改代码
→ 收尾时做验证、复盘、日志与知识回填
```

如果属于新版本增量或大范围功能扩展，默认进一步细化为：

```text
先出系统需求说明书
→ 再出页面清单、规则清单、UI 预览
→ 页面确认后再出数据库架构设计与 API 接口清单
→ 再进入实施文档与代码开发
```

---

## 3. 这套体系解决什么问题

它主要解决 5 类常见问题：

1. 需求还没定就直接开始写代码。
2. 需求、UI、技术实现和审核意见混在同一轮里。
3. AI 探索出了新结论，但只留在聊天里，没有沉淀。
4. 做完以后说不清改了什么、为什么这么改、边界在哪里。
5. 文档越来越多，但越来越难找到真正该看的那份。

## 6. 当前项目的 5 条核心规则

1. 先查文档，文档不足时才探索代码或框架。
2. 探索出的新结论必须回填正式文档。
3. 没有已确认实施文档，不进入正式代码修改。
4. 模块默认先做 UI 预览与确认，再进入后端开发。
5. 正式文档采用规范化表达，不直接搬运聊天原话。

补充默认规则：

6. 每次关键任务开始时，默认必须自动选择 `1` 个最合适的主 agent 先承担主工作，主线程负责收口、集成与对外沟通。
7. 如果新版本需求会影响原有正式页面，确认前先做本地预览页，不直接改已联调页面。
8. 预览页默认按目标正式页面形态承接评审，优先保留正式页核心语义并只承接已确认增量升级，不额外堆放规则说明型提示块；页面确认后，再进入数据库架构设计与 API 接口清单。
9. 需求、页面、规则、实施、数据库、接口、测试结论和代码等关键任务或关键产出，默认都要在主 agent 完成后，再自动选择 `1` 个最合适的独立复核 agent 做第二轮检查，避免遗漏、冲突和不可实施项直接流入下一阶段。
10. 独立复核不是形式动作，复核结论至少要回答“是否遗漏、是否冲突、是否可实施、是否允许进入下一阶段”；仅纯错别字、纯链接修复、纯排版整理这类无语义改动，可不强制走完整双 agent 流程。

## 7. AI 在项目里的默认工作方式

AI 在本项目里不是“直接改代码的黑盒”，而是受文档与门禁约束的执行者。

默认工作方式：

1. 先查索引和正式文档。
2. 用户提出修改意见时，先做一轮需求审查。
3. 如果是新版本任务，先形成系统需求说明书，再推进页面清单、规则清单和 UI 预览。
4. 在文档、数据库设计、接口设计、UI 设计、测试、部署、排障等关键环节，默认先自动选择合适的主 agent 处理对应工作。
5. 如果会影响原有正式页面，先新增本地预览页承接确认，不直接改已联调页面。
6. 预览页默认贴近目标正式页面形态，优先保留正式页核心语义，只承接已确认增量升级；如存在暂未对齐的差异，必须显式标注。
7. 在已确认范围内先完成 UI 预览与收口；页面确认后，再进入数据库设计、接口设计和后续实施文档。
8. 需求文档、页面方案、数据库设计、API 清单、实施文档、测试结论和代码实现等关键产出形成后，默认再自动调取 `1` 个独立复核 agent 做检查，再决定是否进入下一阶段。
9. 只有在实施文档确认后，才进入正式代码开发。
10. 开发时自动选择合适的主开发 agent。
11. 开发后自动选择合适的审查 agent。
12. 任务超过一定复杂度或时长时，补做效率复盘和日志沉淀。

---

## 8. 当前项目为什么要这样做

因为当前项目不是一次性原型，而是一个要持续推进、持续复用、持续交接的软件系统。

这意味着项目必须同时满足：

- 范围清晰
- 边界清晰
- 过程可追溯
- 知识可积累
- 后续可复用

所以当前体系优先保证“稳定推进”，而不是“最快开始写代码”。

## 10. 维护规则

本文件只在以下情况更新：

1. 项目整体运作方式发生变化。
2. 根入口、一级索引或总体查阅方式发生变化。
3. 对外介绍口径需要调整。
