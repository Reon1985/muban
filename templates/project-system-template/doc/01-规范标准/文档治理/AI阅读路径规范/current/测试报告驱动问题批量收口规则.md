# 测试报告驱动问题批量收口规则

## 1. 文档定位

本文件用于定义“测试报告驱动的问题修复”在当前项目中的默认推进规则。

适用场景：

1. 某轮修复工作明确来自测试报告、联调报告、QA 缺陷清单或提交级测试结论
2. 本轮修复完成后需要发版到目标环境验证
3. 希望尽可能减少中途反复发版和碎片化联调

本文件属于长期规则，不是单次任务实施文档。

## 2. 核心规则

当前项目对“测试报告驱动的问题修复”默认采用以下模式：

```text
冻结本轮问题清单
→ 本地一次性修复与自测
→ 统一发版
→ 按测试报告整轮统一验证
→ 回填测试报告、实施文档、升级日志
```

一句话规则：

- 默认批量收口、统一发版、统一验证；例外必须显式豁免。

---

## 3. 执行规则

### 3.1 先冻结本轮问题清单

开始修复前，必须先把测试报告中的问题按本轮口径冻结。

至少分为以下四类：

1. 本轮必修复
2. 延后处理
3. 无法复现
4. 已完成无需重复处理

强约束：

1. 未冻结问题清单，不进入本轮代码修复
2. 中途新发现的问题不默认混入本轮，除非和本轮修复强相关
3. 如果范围扩大到超出原测试报告，应先更新实施文档再继续

### 3.2 本地先收口，不边修边发版

本轮问题默认先在本地完成集中修复和集中自测，再进入服务器发版。

默认不允许：

1. 为同一份测试报告中的零散问题反复多次发版
2. 每修一个问题就上线验证一次
3. 在尚未完成本轮必修复项时反复打断当前批次

### 3.3 发版前必须有统一验证计划

正式发版前，实施文档中至少要明确：

1. 来源测试报告
2. 本轮问题清单
3. 冻结问题清单
4. 统一发版验证计划
5. 回归范围

如果这些字段不完整，不应进入正式发版。

### 3.4 发版后必须整轮验证

发版完成后，不能只抽测单个问题点。

至少要按测试报告逐条验证：

1. 本轮必修复项
2. 关联回归项
3. 发版链路影响到的关键页面或接口

验证完成后，必须同步更新：

1. 测试报告结论
2. 实施文档结论
3. 升级日志

---

## 4. 例外豁免规则

以下情况允许不按“批量收口、统一发版、统一验证”执行：

1. 安全漏洞
2. 生产阻断
3. 数据错误仍在扩大
4. 明确的高优先级线上故障

但必须在实施文档中显式写明：

1. 是否适用本规则
2. 如不适用，豁免原因是什么
3. 为什么不能等到批量发版

一句话：

- 允许例外，但不允许不留痕的例外。

---

## 5. 与 SOP、模板和 hook 的关系

1. 本规则负责定义长期默认口径
2. SOP 负责定义具体怎么执行
3. 模板负责把必填字段固化进实施文档
4. hook / 门禁脚本负责在高风险发布动作前提醒或阻断

边界要求：

1. hook 只能检查字段是否齐备，不代替业务判断
2. 例外是否成立，仍由实施文档和人工确认裁决
3. 自动学习只收集候选经验，不自动修改这条规则

---

## 6. 完成标准

这条规则被视为已落地，至少满足：

1. 已有长期规则正文
2. 已有可直接执行的 SOP
3. 已有可复用模板
4. 已接入发版前门禁提醒或阻断
5. 后续测试报告驱动修复默认按本规则执行
