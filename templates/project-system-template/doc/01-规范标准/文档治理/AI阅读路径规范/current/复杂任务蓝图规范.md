# 复杂任务蓝图规范

## 1. 文档定位

本文件用于定义复杂任务在当前 `__PROJECT_NAME__ 2.0` 体系中的标准推进蓝图。

本文件回答的是“复杂任务应按什么骨架推进”，不负责替代某一次任务的需求正文、页面方案、数据库设计、API 清单或实施文档。

## 2. 适用范围

以下任务默认应套用本蓝图，而不是临场发挥：

1. 横跨需求、页面、数据库、API、实施、测试多个阶段的任务
2. 涉及前后端联动、发版、联调和真实点击验证的任务
3. 需要跨会话、跨人、跨 agent 持续推进的任务
4. 需求边界复杂、返工成本高、存在多个门禁点的任务

以下场景可不强制套完整蓝图：

1. 纯错别字、纯链接修复、纯排版整理
2. 不改变语义和范围的轻量文档修订

---

## 3. 标准蓝图结构

每个复杂任务至少要包含以下 `8` 个要素：

1. 任务定位
   - 当前目标
   - 所属版本
   - 当前阶段
   - 已知约束
2. 阶段清单
   - 需求确认
   - 页面规则 / 预览确认
   - 数据库 / API 设计
   - 实施文档
   - 代码实现
   - 测试与发布
   - 文档回填
3. 每阶段输入
4. 每阶段输出
5. 每阶段门禁
6. 每阶段复核点
7. 风险与回退点
8. 收尾与回填要求

---

## 4. 阶段推进最小要求

### 4.1 需求确认阶段

最小输入：

1. 当前版本入口
2. 已确认需求范围
3. 已知问题与限制

最小输出：

1. 正式需求说明
2. 页面清单 / 规则清单
3. 未决事项边界

门禁：

1. 需求边界未定时，不进入实现

### 4.2 页面规则 / 预览确认阶段

最小输入：

1. 已确认需求文档
2. 既有正式页面或目标承接页面

最小输出：

1. 页面规则
2. 预览页或承接方案
3. UI 差异确认结果

门禁：

1. 页面未确认，不进入数据库 / API 设计

### 4.3 数据库 / API 设计阶段

最小输入：

1. 已确认页面规则
2. 现有表结构与接口基线

最小输出：

1. 数据库设计文档
2. API 接口清单
3. 兼容性与复用判断

门禁：

1. 设计未确认，不进入实施文档和代码

### 4.4 实施阶段

最小输入：

1. 已确认数据库设计
2. 已确认 API 设计

最小输出：

1. 实施文档
2. 范围说明
3. 风险说明

门禁：

1. 无实施文档，不改代码

### 4.5 代码、测试与发布阶段

最小输入：

1. 已确认实施文档
2. 当前环境与发版路径

最小输出：

1. 代码变更
2. 测试结论
3. 发布记录

门禁：

1. 关键产出需独立复核
2. 测试未完成，不作完成结论

### 4.6 收尾与回填阶段

最小输出：

1. 规则回填
2. SOP 回填
3. 升级日志或问题总结

---

## 5. 标准执行方式

复杂任务默认按以下顺序推进：

```text
入口文档确认
→ 需求与规则确认
→ 页面预览确认
→ 数据库 / API 设计
→ 实施文档
→ 代码实现
→ 测试 / 发版 / 联调
→ 文档回填与升级日志
```

执行要求：

1. 每进入下一阶段前，必须确认上一阶段产出已成基线
2. 关键产出默认采用“主 agent + 独立复核 agent”模式
3. 如某阶段发现前一阶段边界错误，应回退修正文档，不允许硬推进

---

## 6. 默认边界

本规范允许：

1. 为复杂任务建立统一推进骨架
2. 统一阶段、门禁、复核点和收尾口径

本规范不负责：

1. 替代某次任务的业务事实判断
2. 替代版本包中的正式需求和实施文档
3. 自动放行任何阶段

---

## 7. 完成标准

一个复杂任务只有在以下条件全部满足时，才可视为按蓝图完成：

1. 每个阶段都有明确输入输出
2. 关键门禁没有被跳过
3. 关键产出经过独立复核
4. 测试、发布和回填已经完成
