# 轻量 hook 清单设计规范

## 1. 文档定位

本文件用于定义当前项目中“轻量 hook 清单”应如何设计、如何落地、如何控制边界。

本文件回答的问题是：

1. 哪些动作适合做 hook
2. 哪些动作不适合做 hook
3. hook 应输出什么
4. hook 和正式规则、SOP、自动学习分别是什么关系

本文件是长期规则文档，不是某次任务的临时脚本说明。

## 2. 核心结论

当前项目只允许“轻量 hook”进入正式体系。

轻量 hook 的默认职责只有三类：

1. 提醒
2. 检查
3. 收集

一句话：

- hook 只自动化机械动作，不自动化正式判断。

---

## 3. 适合进入 hook 的动作

以下动作优先考虑纳入 hook 清单：

1. 进入任务前检查是否已读入口文档
2. 改代码前检查是否已有实施文档
3. 任务结束前提醒是否需要回填
4. 关键产出后提醒是否需要独立复核
5. 长任务结束后提醒是否需要效率复盘
6. 可结构化收集的候选经验整理
7. 高风险发版命令执行前，检查当前活跃实施文档是否满足测试报告批量收口门禁

这些动作的共同特征是：

1. 高频重复
2. 判断标准相对稳定
3. 不依赖复杂业务语义
4. 出错后不会直接污染正式规则

---

## 4. 不适合进入 hook 的动作

以下动作默认不应纳入 hook：

1. 自动批准进入下一阶段
2. 自动改变阶段门禁
3. 自动修改正式规则文档
4. 自动修改 skill
5. 自动给出最终评审结论
6. 自动覆盖用户刚刚确认过的业务边界

一句话：

- 只要动作会改变未来默认执行方式，就已经超出轻量 hook 的边界。

---

## 5. hook 的标准输出

每个 hook 只允许输出以下类型之一或组合：

1. 提醒信息
2. 检查结果
3. 待补项列表
4. 候选经验条目

不允许输出：

1. 直接生效的正式结论
2. 自动写入的长期规则
3. 自动切换的流程状态

---

## 6. hook 清单的记录方式

hook 清单本身应以“文档先行”方式维护。

每条 hook 候选最少记录：

1. 名称
2. 触发时机
3. 目标动作
4. 输出类型
5. 边界说明
6. 是否已试运行

如果这些信息不完整，不应直接进入实施。

---

## 7. 与 SOP 和自动学习的关系

1. SOP 负责写“人应该怎么执行”
2. hook 负责提醒“现在是不是该执行某个动作”
3. 自动学习负责“本轮有哪些经验值得作为候选被收集”

边界要求：

1. hook 可以提醒回填，但不代替回填
2. hook 可以收集候选，但不代替审查
3. hook 可以触发检查，但不代替最终裁决

---

## 8. 进入实施前的门槛

某条 hook 候选要进入真实实施，至少满足：

1. 已在文档中写清触发条件
2. 已在文档中写清边界
3. 已完成至少一轮试运行
4. 试运行中没有出现越权行为
5. 对应正式文档和执行清单已有稳定落点
6. 如果 hook 要对高风险发布动作做阻断，必须已有稳定的长期规则、SOP 和实施模板作为依据

---

## 9. 完成标准

轻量 hook 清单机制建立完成，至少满足：

1. 已定义什么适合做 hook
2. 已定义什么不能做 hook
3. 已定义 hook 的最小记录字段
4. 已定义进入真实实施前的门槛
