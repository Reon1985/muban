# 开发规范

## 1. 文档定位

本文件用于定义本项目进入“代码开发阶段”后的统一开发规范。

适用对象：

- AI 开发执行者
- 开发经理
- 前端开发
- 后端开发
- 联调与测试成员

本文件是项目级长期规则文件，和 [WORKFLOW.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/WORKFLOW.md)、[AGENTS.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/AGENTS.md) 配套使用。

边界说明：

- [AGENTS.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/AGENTS.md) 负责定义角色模型、主导关系和协作边界
- 本文件负责定义开发阶段的阅读顺序、执行门禁、agent 选型和收尾细则
- [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md) 负责 agent / skill 的名称说明和推荐选型参考

它重点回答四个问题：

1. 开发前必须先读哪些文档
2. 开发过程中应遵守哪些规则
3. 如果产生新的开发规范，应该写回哪里
4. 开发时和开发后，应该如何自动选择合适的 agent

---

## 2. 总原则

本项目默认采用以下开发链路：

```text
先读文档入口链与上下文聚合
→ 如属大型跨会话任务，再补读 `.project_state.json` / `.handoff/` 等连续性资产
→ 确认实施方案已获批准
→ 自动选择合适的开发 agent
→ 在批准范围内开发
→ 如果产生新开发规范，回填文档
→ 自动选择合适的审查 agent
→ 代码审查
→ 再进入验证、效率复盘、日志与收尾
```

一句话规则：

- 文档入口优先，关键任务双 agent 协作，连续性资产按需补充。

补充规则：

- 模块默认采用“先 UI、后后端”的双阶段推进方式
- 先把需求、菜单结构、页面规则和 UI 确认下来，再进入数据库、接口和后端开发
- 写文档、数据库设计、接口设计、页面 UI 设计、测试等各个环节，默认优先选择 `1` 个最合适的主 agent 承担主工作，主线程负责收口与集成
- `.project_state.json`、`.handoff/`、`TraceID` 等机制适合大型、多阶段、跨会话任务；共享模板中默认视为可选增强层，不作为所有项目的强制前置

## 3.1 代码定位优先顺序

进入代码修改前，默认先按下面顺序缩小阅读范围：

```text
项目文档总览
→ 当前目录 README
→ 前端.md / 后端.md / 权限架构说明
→ 目标模块目录
→ 目标文件
→ 仍不足时再扩大探索
```

执行规则：

1. 改页面优先看 [前端.md](__PROJECT_ROOT__/doc/00-入口/项目定位与环境/前端.md)。
2. 改接口优先看 [后端.md](__PROJECT_ROOT__/doc/00-入口/项目定位与环境/后端.md)。
3. 改用户 / 部门 / 岗位 / 角色 / 菜单 / 数据权限时，优先补读 [框架用户组织角色数据权限架构说明.md](__PROJECT_ROOT__/doc/01-规范标准/后端/权限与数据范围规范/current/框架用户组织角色数据权限架构说明.md)。
4. 文档已经明确给出入口文件时，不再默认全量遍历无关目录。
5. 只有当文档无法支撑判断时，才扩大到全库搜索。

### 3.1.1 实施门禁最小判断

进入正式改码前，至少要明确回答下面这些问题：

1. 是否已经完成入口链和文档阶段定位。
2. 是否已经确认当前任务范围和不改项。
3. 是否已经确认实施文档批准状态。
4. 当前变更是否确实落在已批准范围内。

如果上述任一项不能明确回答，默认不进入正式开发。

补充检查：

1. 是否绕过了入口链直接扫代码。
2. 是否在没有批准实施文档的情况下提前进入开发。
3. 是否把当前项目事实误当成可直接上提的共享长期规则。
4. 是否需要先补充项目级例外声明，再继续实施。

## 3.2 模块默认推进方式

从当前版本开始，模块型工作默认采用双阶段推进：

```text
阶段 A：详细需求确认 → 菜单结构 / 页面规则确认 → 前端 UI 预览 → 高频调整 → UI 确认
阶段 B：数据库 / 接口 / 后端实施文档确认 → 后端开发 → 前后端联调 → 验证收口
```

执行规则：

1. 先把页面和交互看对，再做该模块后端。
2. 每次用户提出需求修改意见时，先按讨论内容动态选择合适的需求审查 agent，协助审视需求的合理性、完整性、边界和影响，再进入 UI 调整或后续开发。
3. UI 未确认前，不进入该模块的数据库表结构、接口契约和业务服务正式开发。
4. UI 阶段允许频繁调整，但必须限定在当前模块、当前已确认的菜单结构和页面规则范围内。
5. 如果 UI 调整导致菜单结构、页面规则、交互流程或实现范围发生实质变化，必须更新实施文档并重新确认。
6. 在同一份已批准实施文档覆盖的 UI 范围内，可连续做多轮前端微调，不要求每次微调都新建升级日志。
7. 一旦 UI 确认，后续后端开发必须以已确认 UI 为依据，避免再次反向重做页面。

## 3.2.1 全工作 agent 强制规则

从当前版本开始，项目内所有工作默认都必须先匹配合适的主 agent，再进入执行。

适用范围：

1. 需求讨论与需求审查
2. 文档编写、整理、迁移、评审
3. 页面 UI 设计与交互方案设计
4. 前端开发
5. 后端开发
6. 数据库表结构设计与调整
7. 前后端接口设计与对齐
8. 浏览器测试、接口测试、回归测试、验收测试
9. 部署、发版、联调、排障
10. 复盘、治理规则更新、交接单整理

执行规则：

1. 每一项工作开始前，默认先选 `1` 个最合适的主 agent。
2. 需求、页面、规则、实施方案、数据库架构、API 清单、测试结论和代码实现等关键产出形成后，默认再选 `1` 个独立复核 agent 做审查；主 agent 不作为默认复核 agent。
3. 如果任务明显跨多个维度，可按需追加 `0~2` 个补充 agent，但必须保留 `1` 个主 agent。
4. 主线程负责下达任务边界、跟踪关键路径、合并结论和最终收口。
5. 主线程不再把“自己直接做”当默认起点；默认起点是“先判断由哪个 agent 主做最合适”。
6. 只有以下 `3` 类情况，才允许不启用 agent：
   - 当前平台上确实没有合适 agent
   - agent / 平台能力当下不可用，继续等待只会阻塞任务
   - 用户明确要求本轮不要使用 agent
7. 即使属于上述例外，主线程也应按“本来应该由哪类 agent 负责”的口径组织思路，不能退回无结构硬做。

默认理解：

- 这条规则不是“每件事都堆很多 agent”，而是“每件事都先找最合适的 agent 来主做”。
- 项目默认协作模型统一为“`1` 个主 agent + 主线程收口”，而不是“主线程硬扛，agent 只是可选补充”。
- 对关键产出而言，默认协作模型进一步固定为“`1` 个主 agent + `1` 个独立复核 agent + 主线程收口”。
- 该规则覆盖编码和非编码环节，不再只限于正式开发阶段。

### 3.2.1.1 关键产出的独立复核规则

从当前版本开始，以下关键产出在主 agent 完成后，默认必须再经过 `1` 个独立复核 agent 审查后，才能作为本轮正式结论继续向下推进：

1. 需求说明书、页面清单、规则清单
2. UI 方案、页面结构设计、预览页方案
3. 数据库架构设计、表结构设计
4. API 接口清单、接口契约和技术方案
5. 实施文档、评审文档、测试结论
6. 代码实现与代码修正结果

非代码产出的复核重点：

1. 是否遗漏功能点、页面入口、字段口径、角色范围、边界条件或异常场景
2. 是否存在规则冲突、术语不统一、跨文档不一致或与既有规范脱节
3. 是否具备完整性、全面性、可实施性和前后衔接关系
4. 是否符合项目文档规范、UI 规范、架构规范、接口规范和版本口径

代码产出的复核重点：

1. 是否正确实现已确认方案
2. 是否保持代码简洁、可维护、可扩展，避免临时堆叠式实现
3. 是否符合既有架构、命名、边界和复用规范
4. 是否存在回归风险、范围漂移、缺失测试或缺失文档回填

强约束：

1. 复核 agent 默认不与主 agent 完全重合。
2. 未完成独立复核的关键产出，只能视为阶段性结果，不作为最终收口版本。
3. 如果复核指出遗漏、冲突或不可实施项，必须先修正，再进入下一阶段。

轻量例外：

以下改动默认不强制走完整“主 agent + 独立复核 agent”流程，但仍应完成主线程自检：

1. 错别字、标点、纯措辞顺滑化调整
2. 链接失效修复、目录跳转修复
3. 不改变业务含义、规则口径、结构边界的纯排版整理

只要改动触及以下任一内容，仍必须走独立复核：

1. 需求范围、功能点、页面规则、字段口径
2. 页面结构、交互行为、权限边界、校验条件
3. 数据库结构、接口契约、实施边界、测试结论
4. 任何可能影响后续开发、联调、评审或回切的内容

复核结论要求：

每次独立复核后，至少要明确输出以下结论：

1. 是否存在遗漏项
2. 是否存在冲突项或与既有正式文档不一致
3. 当前结果是否具备可实施性 / 可继续推进性
4. 是否允许进入下一阶段；如不允许，应说明先修什么

非代码产出的固定检查清单：

1. 完整性：功能点、页面、字段、角色、异常场景是否漏项
2. 一致性：是否与需求、页面规则、实施文档、版本口径和既有正式文档一致
3. 可实施性：是否能直接指导后续 UI、数据库、接口、测试或开发动作
4. 边界清晰度：本轮做什么、不做什么、依赖什么、风险在哪里是否明确
5. 规范符合度：是否符合文档规范、UI 规范、架构规范、接口规范和命名口径
6. 返工风险：是否存在会在后续造成回切、联调、评审或发布返工的高风险点

复核视角优先规则：

1. 需求 / 页面 / 规则类产出，复核优先选与主 agent 不同的业务、设计或结构视角
2. 数据库 / 接口 / 技术方案类产出，复核优先选架构、性能、边界控制视角
3. 代码实现类产出，复核优先选代码审查、架构审查、安全审查或 UI 一致性视角
4. 不要让主 agent 只换一个近似身份就再次审自己

### 3.2.1.2 子 agent 等待窗口规则

从当前版本开始，只要已经启动了子 agent，默认等待策略统一按“长窗口优先”执行，不再做高频短轮询。

执行规则：

1. 只要子 agent 承担的是实质性任务，首次等待窗口默认 `10` 分钟起步。
2. 任务越重，首次等待窗口越长；不要还没到合理完成时间就频繁 `wait`。
3. 只有极轻量的探路、单点查证、几分钟内可完成的小问题，才允许使用短于 `10` 分钟的等待窗口。
4. 如果子 agent 的结果不在当前关键路径上，主线程优先继续做明确不重叠的本地工作，不为了“看看有没有回”去做碎片化轮询。
5. 如果子 agent 承担的是构建、测试、联调、批量验证、较大代码实现或长文档整理，默认把等待窗口提升到 `15~30` 分钟，再视任务复杂度继续增加。
6. 如果子 agent 出现平台级故障、结果流中断或明显超出合理完成时间，主线程先判断是否切回本地收口，而不是无限追加等待。
7. 如果同一关键路径上连续出现 `stream disconnected`、`429`、`503` 等平台级故障，默认停止串行重试多个子 agent；主线程先完成本地实现、自检、编译或验证，再在结果稳定后补 `1` 次替代复核。

推荐基线：

1. 轻量探路类：`3~10` 分钟
2. 常规实现 / 分析类：`10~15` 分钟
3. 构建 / 测试 / 联调类：`15~30` 分钟
4. 大范围实现 / 大文档整理类：`30` 分钟以上

一句话：

- 子 agent 默认“等够一个合理完成窗口”，不是“隔几十秒去问一次”。

## 3.2.2 原页面改造的本地预览优先规则

从当前版本开始，如果新版本需求会影响已经联调或已发布的正式页面，默认先走“本地预览页”而不是直接修改原页面。

标准顺序：

```text
系统需求说明书
→ 页面清单 / 规则清单
→ 本地预览页 / 预览路由
→ UI 确认
→ 数据库 / API 设计
→ 实施文档
→ 正式页面改造
```

执行规则：

1. 页面确认前，不直接改原有正式页面结构和正式接口调用链路。
2. 本地预览页应使用独立路由、独立假数据或隔离态数据装配，避免影响现有已跑通页面。
3. 本地预览页的目标不是做一次性草图，而是尽量贴近最终正式信息架构，降低后续回切成本。
4. UI 确认后，再决定是将预览页能力合并回原页面，还是以新正式页面替换原页面。
5. 如果预览阶段发现页面结构、交互边界或实现范围发生明显变化，必须先更新需求或实施文档，再进入正式开发。
6. 如果预览对象是既有正式页面，优先单独补一份对应 UI 正文或实施文档，不把“新增业务页面方案”和“既有页面风格预览方案”混写在同一份正文里。
7. 既有正式页面的预览页默认保留正式页的搜索项、核心字段和操作语义，并以“正式页保留 + 已确认增量升级”为原则承接新需求；如本轮本身包含视觉升级，也应以正式页面后续可回切为前提，不做脱离正式页语义的全新样式试验。
8. 预览页默认直接按目标正式页面形态展示，不额外增加规则说明型提示块、页面说明副标题或为评审临时拼接的大段解释文案；确需保留说明时，只允许保留最小必要差异提示，并同步写入实施文档。
9. 既有正式页面做预览重构前，默认先抽取正式页的“搜索项 / 核心字段 / 操作语义 / 特殊校验口径”四项清单，再进入页面实现，避免边做边回查导致语义偏移。
10. 如果预览页未完全对齐正式页的确认弹窗、禁用条件、引用校验范围或触发入口，必须在页面文案和实施文档中显式标注“不对齐”，不能直接沿用正式动作文案误导评审。
11. 当同一预览体系的页面数超过 `8` 个，或同时覆盖“列表页 / 详情页 / 新增业务页”三类页面时，必须补一个统一预览总览页或分组导航入口，优先在本轮早期落地，避免菜单平铺导致评审、联调和回归成本持续上升。
12. 统一预览总览页负责分组浏览、评审说明和默认落地页，不替代各预览页的直达路由；当详情型预览页已经明显挤占菜单层级时，可以收口为隐藏路由，但必须保留直达访问与正确的 `activeMenu` 回指。

## 4. 开发实施基本规则

### 4.1 只在批准范围内开发

开发必须以已确认的实施文档为边界。

不允许：

- 趁开发顺手扩功能
- 顺带改无关模块
- 未经确认扩大页面或接口范围

### 4.2 先复用已有底座，再新增实现

在已有框架能力可承载的前提下，优先复用：

- 现有 layout
- 现有组件
- 现有接口
- 现有组织、权限、审批、日志、文件能力

新增实现必须说明“为什么现有底座不够”。

### 4.2.0 不要兜底式开发

当需求、字段、接口、权限、版本口径或项目事实仍不明确时，默认动作不是“先写一个兜底实现再说”，而是：

1. 先回到正式文档确认口径。
2. 先确认是否缺实施文档、缺例外声明或缺项目事实说明。
3. 先明确哪些内容已经确认，哪些内容仍未确认。
4. 结论明确后，再进入正式实现。

默认不允许：

1. 用猜测字段、猜测接口、猜测权限直接完成正式实现。
2. 用临时假数据、预览数据或硬编码常量伪装正式交付结果。
3. 用“先兼容一下”“先兜底跑通”代替正式边界判断。
4. 把只在当前会话里说清的结论，当成长期规则直接写进代码。

如果确实需要临时兼容或临时兜底，必须同时满足：

1. 实施文档里已经显式写明原因、范围和退出条件。
2. 不影响当前正式主链语义。
3. 收尾时会同步回填文档和升级日志，不让临时方案长期悬空。

### 4.2.1 需求修改意见的前置审查规则

当用户提出以下任一类型的修改意见时：

- 需求增删改
- 菜单结构调整
- 页面结构调整
- 交互方式调整
- 权限或数据范围口径调整
- 模块边界或实现范围调整

默认动作不是直接改页面或改代码，而是：

1. 先判断本次修改主要涉及业务规则、页面结构、交互方式、模块边界还是实现范围
2. 默认选择 `1` 个最合适的主需求审查 agent
3. 如果本次修改同时跨两个以上维度，或存在明显边界、依赖、冲突风险，再按需追加 `0~2` 个补充审查 agent
4. 先审查该修改意见的合理性、完整性、边界和潜在遗漏
5. 先与用户完成一轮聚焦讨论
6. 结论明确后，再进入 UI 修改、实施文档更新或代码开发

补充口径：

- 该规则是需求审查前置，不改变现有文档分级更新规则
- 高频 UI 微调也适用，但只需做轻量需求审视，不要求每次都产出独立正式文档
- 需求审查 agent 总数默认不超过 `3` 个，避免把轻量确认做成重型会审
- 多 agent 参与时，必须明确 `1` 个主审查 agent，其余 agent 只补充单一视角，由主线程统一收口

### 4.2.2 主数据关联与编码设计规则

从当前版本开始，只要新需求会新增表、调整表结构或新增跨表引用关系，默认同时遵守以下“稳定引用键”规则：

1. 新表默认只存“稳定引用键 + 本表新增业务信息”，不再为了展示方便冗余存储主数据名称、简称、组织名称等可变文本。
2. 如果被引用主数据已经存在稳定业务编码 `code`，优先用 `code` 参与跨模块识别、接口传递、导入导出和人工核对。
3. 如果当前底座还没有统一业务编码，则使用主键 `id` 作为稳定引用键；不能因为缺少 `code` 就退回去存名称字段。
4. 展示名称、简称、组织名、岗位名等文本一律在查询/返回阶段按引用键实时装配，不作为新表默认落库内容。
5. 只有在“必须保留历史快照”的明确场景下，才允许单独设计名称快照字段，并且要在设计文档中写清原因、快照时点和回写规则。
6. 业务校验、删除校验、汇总计算和权限判断，必须基于稳定引用键，不得基于名称文本。
7. 如果现有旧表已经存在名称冗余，新需求默认不继续扩散这套做法；本轮新增表按新规则执行，旧表是否收敛单独评估。

针对当前项目的落地口径：

- 已有稳定业务编码的主数据，对外展示优先返回 `code`
- 尚未建立统一业务编码的主数据，新增表先用主键 `id` 做稳定引用键
- 新表建模时，优先判断“这个字段是不是本表新增事实”，如果不是，默认不要入库

### 4.2.3 前端 TypeScript 门禁规则

从当前版本开始，前端静态类型门禁按以下规则执行：

1. `pnpm ts:check` 的目标是检查真实类型错误，不承担“未使用变量/参数”清理职责。
2. `TS6133`、`noUnusedLocals`、`noUnusedParameters` 这类噪音，统一交给 ESLint 处理，不再作为 `vue-tsc` 的主阻塞项。
3. 共享层类型错误、缺失声明、store / hook / util 上游阻塞，必须优先修复，避免页面层被连锁误伤。
4. 对历史演示页、低代码页、重型设计器页，如果短期内无法在本轮安全完成类型收口，允许定点使用 `ts-nocheck`，但必须满足：
   - 仅限遗留页或非主链路页面
   - 必须在升级日志中说明原因
   - 不得把 `ts-nocheck` 扩散到当前主业务链路
5. 每次大范围前端改动收尾前，默认复跑一次项目定义的类型检查命令。
6. 如果类型检查因内存不足而中断，默认先使用项目约定的高内存重跑方式复跑，再判断是否存在真实类型错误。
7. 如果 `ts:check` 首先报错集中在 `src/types/auto-components.d.ts` 这类生成声明文件，并且表现为大批量语法错误，应先判断是否为生成文件污染或损坏，不要直接把问题归因到当前业务改动。

当需求、页面、接口或数据库设计涉及主数据对象时，默认先判断该对象是否需要稳定识别标识。

主数据对象通常包括但不限于：

- 客户
- 供应商 / 合作方
- 公司主体
- 员工
- 项目
- 合同
- 发票
- 其他会被多个模块重复引用的核心业务对象

默认规则：

1. 数据库内部关联优先使用主键 `id`，不直接以名称字段作为物理关联键。
2. 跨模块识别、导入导出、接口传递、对账或人工沟通场景，优先设计并使用稳定的业务编码 `code`。
3. 展示名称 `name` 默认作为可维护字段，不因“被下游引用”而直接定义为不可修改。
4. 如果历史单据需要保留当时名称，应增加名称快照字段，而不是依赖名称不可修改来保证历史一致性。
5. 只有在业务上确实存在法定名称锁定、合同文本锁定、审计锁定等明确原因时，才单独定义“名称不可修改”规则，并写清锁定原因、适用范围和生效时点。

推荐字段分层：

- `id`：系统内部主键，供数据库关联使用
- `code`：业务编码，供跨模块识别、导入导出、接口传递和线下沟通使用
- `name`：展示名称，供页面展示和日常维护使用

适用判断：

- 如果对象会被多个模块反复引用，默认应设计业务编码
- 如果名称存在更名可能，默认不得让名称承担关联职责
- 如果对象只属于中间表、日志表、临时表、纯明细子表，可不单独设计业务编码，必要时采用“主单编码 + 行号”的方式

### 4.2.4 测试报告驱动修复的批量收口规则

从当前版本开始，凡是本轮修复明确来源于测试报告、联调报告或 QA 缺陷清单，默认都按“批量收口、统一发版、统一验证”推进。

执行规则：

1. 先冻结本轮问题清单，再进入代码修复。
2. 本轮必修复项未完成前，不默认反复发版到服务器。
3. 发版前，实施文档必须写明来源测试报告、本轮问题清单、冻结问题清单和统一发版验证计划。
4. 发版后，必须按测试报告逐条统一验证，不得只抽测单个点位。
5. 安全漏洞、生产阻断、数据错误持续扩大等紧急场景允许例外，但必须在实施文档中写明豁免原因。
6. 发版前默认通过仓库级门禁脚本检查该规则是否已在当前活跃实施文档中落地。

### 4.3 开发时同步维护文档一致性

如果开发过程中发生以下情况，必须同步补文档：

- 实施范围发生细化
- 代码路径与原方案不同
- 反复出现的通用页面问题被识别出来
- 页面设计类共性规则被新确认
- 页面检查口径或验收口径被新确认

### 4.3.1 页面规则与检查项回填规则

从当前版本开始，凡是页面开发、页面调整、UI 收口、联调验收过程中沉淀出的“可复用共性结论”，不得只停留在会话里，必须按类型写回正式文档。

回填原则：

1. 页面布局、字段显隐、按钮文案、时间展示、只读规则、候选范围规则等长期页面口径，写回 `01-规范标准/前端/UI规范/versions/...`
2. 易错点、重复 Bug、联调中高频踩坑、Avue 配置误用、i18n 文案错误、重复按钮等问题，写回 `01-规范标准/前端/页面实现规范/current/前端开发常见错误检查清单.md`
3. 与某个模块强相关、但会长期影响页面实现定位的口径，写回对应模块定位文档，例如 `00-入口/项目定位与环境/前端.md`

执行要求：

1. 每次正式页面改动结束前，至少判断一次“本轮有没有新增通用规则或通用错误”
2. 如果有，必须在同一轮任务内完成文档回填，不允许留到“下次顺手”
3. 后续做页面 UI 需求分析、页面实现、页面联调、页面验收时，必须优先读取这些长期文档，而不是重新从会话历史猜规则
- 发现框架真实机制和原判断不一致
- 新增了可复用的开发规则
- 发现了新的技术限制、风险或坑点

补充执行口径：

- 不是每次小改动后都要全量更新全部文档
- 文档更新按优先级执行，优先保证会影响下一步判断的文档先同步

### 4.3.1 文档更新优先级

开发过程中的文档默认分为三档：

#### `A` 级：必须立刻更新

满足以下任一情况时，必须在当前工作结束前立即更新：

- 当前实施文档的状态、范围、边界已经变化
- 当前模块需求、菜单结构、页面规则、业务口径已经变化
- 新发现会直接影响下一步开发判断的框架结论
- 新发现阻塞型风险、严重问题、关键限制

一句话原则：

- 不立刻更新，就可能导致下一步做错

#### `B` 级：可按半天或一天批量更新

以下文档默认不要求每个小改动后立刻更新，可先记录“待回填清单”，在半天、一天或一次集中收尾时统一更新：

- 升级日志
- 效率提升
- 前端 / 后端说明
- 框架说明类文档
- 全局Agent与Skill说明.md

一句话原则：

- 重要，但不直接阻塞当前开发

#### `C` 级：可按阶段统一更新

以下文档默认在一个模块完成、一个阶段结束或明确收尾时再统一整理：

- README.md
- 项目文档总览.md
- doc目录结构与使用规范.md
- 文档迁移、归档、导航类文档
- 阶段性总结类文档

一句话原则：

- 导航与归档类文档，不参与每轮微调

### 4.3.2 正式文档编写表达规则

正式文档应以规范化表达为主，不直接照搬口语化输入。

默认规则：

1. 文档应基于需求意图进行抽象、整理和归纳，再转换为正式表述。
2. 除会议纪要、原始输入记录、外部原文摘录等场景外，不直接将对话原话写入正式需求、方案、实施或规范文档。
3. 需求文档、方案文档、实施文档和规范文档中，应优先使用项目统一术语、统一结构和统一口径。
4. 如果原始表达存在口语化、临时性、上下文依赖强或边界不清的问题，应先规范化，再写入正式文档。
5. 文档中应直接陈述规则、约束、范围、目标和结论，避免使用“用户希望”“本次沟通认为”等对话式表达作为正式正文主语。

一句话原则：

- 正式文档记录的是经过整理后的规则与结论，不是聊天原话的直接搬运。

### 4.4 不把新规范只留在会话里

如果本次开发过程中形成了新的“可复用开发规范”，不能只留在聊天记录里。

必须按类型回填：

- 项目级开发规则：写入 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md)
- AI / rule / skill 执行机制优化：写入 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)
- agent / skill 说明、选型口径、推荐搭配变化：写入 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)
- 模块级实现约束：写入对应模块正式文档
- 框架认知：写入对应认知文档或方案文档

### 4.5 开发前置效率控制

为减少开发过程中无效验证和收尾遗漏，正式开发前默认补做以下动作：

1. 先定义“本轮目标文件清单 + 定向校验命令”
2. 先列出“收尾需更新的文档清单”，并标记 `A / B / C` 优先级
3. 如果本轮同时包含壳层改造和功能页改造，先拆开验收边界
4. 如果本轮处于 UI 阶段，先明确哪些改动属于“高频 UI 微调”，哪些改动一旦发生就要回到实施文档重新确认
5. 如果本轮涉及后端编译、打包或发版，执行前先同时确认运行时和构建工具版本与项目文档一致；一旦不一致，必须先切到正确工具链再继续

### 4.6 路由壳层与 Tab 改造前置规则

如果本轮改动涉及：

- 顶部菜单 / 左侧菜单跳转
- Tab 工作区复用
- 预览态 / 联调态切换
- 依赖路由 query 的壳层参数

则开发前必须先明确两类口径：

1. 业务页唯一键是什么
2. 哪些 query 属于壳层共享参数

默认规则：

- `path` 或稳定业务标识，才可作为“是否同一功能页”的判断依据
- `_tab` 这类临时 query 只能作为页签实例标识，不能作为业务页唯一键
- `live` 这类壳层共享参数，在菜单切页、Tab 切换、路由纠偏时必须保留
- 验证时不能只测“从菜单点击进入”的路径，还要补测“直接路由进入 / 无 `_tab` 进入 / 带共享参数进入”的路径

### 4.7 结构迁移与物理拆分前置规则

如果本轮任务涉及：

- 目录迁移
- 模块拆分
- 壳层与业务层解耦
- 前后端命名空间调整

则开发前必须先明确以下内容：

1. 旧路径到新路径的迁移清单
2. 哪些外部访问地址保持不变
3. 哪些文件属于“结构迁移范围”，哪些文件明确不动
4. 本轮是否只做骨架拆分，还是同时叠加业务逻辑改造

默认规则：

- 第一轮结构迁移优先只做“骨架拆分”，不叠加新的业务规则改造
- 当目标是“后续可提取复用”时，除了检查全局主链，还必须补查业务首页、快捷入口、共享组件等是否残留反向依赖
- 结构迁移后的默认收尾顺序为：
  - `rg` 检查旧路径残留
  - 定向静态校验
  - 模块间反向依赖检查
  - 代表性访问地址验证
  - 独立审查 agent 复核
- 结构迁移完成后，必须同步更新当前实施文档、项目文档总览、前端 / 后端认知文档和系统升级日志

## 5. 新开发规范回填规则

### 5.1 什么叫”新的开发规范”

以下内容都视为新的开发规范：

- 新的页面壳层接入规则
- 新的路由或菜单改造规则
- 新的组件复用约束
- 新的 API 对接约束
- 新的联调流程要求
- 新的代码审查门禁
- 新的 agent 协作规则
- **新的前端 UI 组件模板或页面类型**
- **新的外键选择规范（如客户/财务组织/人员选择规则）**
- **新的 API 接口映射或接口路径规范**

### 5.2 回填动作

如果本次开发中形成了新的开发规范，默认执行以下动作：

1. 判断该规则是否为”项目级长期规则”
2. 判断它属于 `A / B / C` 哪一档文档更新
3. 如果属于 `A` 级，当前工作结束前立即更新
4. 如果属于 `B` 级，加入当日或当次集中回填清单
5. 如果属于 `C` 级，留待模块或阶段收尾时统一整理
6. 如果是项目级长期规则，补入 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md)
7. 如果同时影响 AI 执行流程或 skill 治理，补入 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)
8. 如果同时影响 agent / skill 选型说明，补入 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)
9. 如果只影响某一模块，补入对应模块文档和升级日志

### 5.3 未完成回填，不算正式收尾

如果本次开发形成了新的规则，但没有写回文档，则该任务只能算”代码完成”，不能算”项目收尾完成”。

## 6. 开发 agent 自动选取规则

### 6.1 总规则

每次正式开发时，默认自动选择 `1` 个最合适的开发 agent 作为主执行者。

原则：

- 一个开发任务只设 `1` 个主开发 agent
- 如需其他 agent 参与，应作为辅助角色，不替代主开发 agent
- 主开发 agent 选择依据是“任务类型”，不是“谁最方便”

### 6.2 推荐选取规则

#### A. 前端页面、交互、壳层、组件开发

优先选择：

- `agency-frontend-developer`

如涉及强 UI 重构或交互统一，可协同：

- `agency-ui-designer`
- `agency-ux-architect`

#### A-1. 用户提出需求修改意见时的前置审查 agent

默认按修改类型选择主审查 agent，并在必要时追加补充 agent：

- 业务需求、范围、功能点合理性：`agency-product-manager`
- 页面结构、交互方式、信息组织：`agency-ui-designer`
- 复杂交互与体验流转：`agency-ux-architect`
- 模块边界、通用能力复用、结构拆分：`agency-software-architect`

执行原则：

- 默认只选 `1` 个最合适的主审查 agent
- 如果修改同时跨业务、UI、交互、架构等多个维度，可追加 `1~2` 个补充 agent
- 需求审查 agent 总数默认不超过 `3`
- 多 agent 参与时，补充 agent 不重复复述全案，只补主审查 agent 未覆盖的风险
- 最终结论由主线程统一合并并对外表达
- 审查完成后，再进入 UI 修改或代码实现

#### B. 后端接口、数据结构、服务边界开发

优先选择：

- `agency-backend-architect`

如涉及复杂实现落地，可协同：

- `agency-senior-developer`

#### C. 跨前后端联动或模块级重构

优先选择：

- `agency-software-architect` 先做结构把关
- 再按主工作面选择 `agency-frontend-developer` 或 `agency-backend-architect` 作为主开发 agent

#### D. 快速原型或高速度验证

优先选择：

- `agency-rapid-prototyper`

但前提是：

- 不能绕过本项目的实施文档门禁
- 原型结论仍需回填正式文档

### 6.3 选择记录要求

只要进入正式开发，建议在任务说明或实施记录中明确：

- 本次主开发 agent 是谁
- 为什么选它
- 本次是否有辅助 agent

说明：

- 具体选型口径优先参考 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)

## 7. 开发后代码审查 agent 自动选取规则

### 7.1 总规则

每次开发完成后，默认自动再选择 `1` 个合适的审查 agent 做代码审核。

要求：

- 审查 agent 不能和主开发 agent 完全承担同一职责
- 审查重点优先看：正确性、影响范围、回归风险、遗漏测试、边界漂移

### 7.2 默认审查 agent

默认优先选择：

- `agency-code-reviewer`

### 7.3 补充审查规则

如果任务有明显专项风险，可追加第二审查视角：

- 架构或跨模块风险：`gstack-review` 或 `agency-software-architect`
- 安全风险：`agency-security-engineer`
- UI 一致性风险：`agency-ui-designer` 或 `gstack-design-review`
- 性能风险：`agency-performance-benchmarker`

### 7.4 审查结论要求

代码审查阶段至少要输出：

- 发现的问题
- 风险等级
- 是否存在范围外改动
- 是否存在缺失文档回填
- 是否建议继续修改后再交付

说明：

- 具体审查 agent 的推荐口径优先参考 [全局Agent与Skill说明.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/全局Agent与Skill说明.md)

## 8. 开发完成后的标准收口

开发完成后，默认按以下顺序收口：

1. 自检
2. 代码审查 agent 审查
3. 必要修正
4. 更新受影响文档
5. 判断本次正式开发累计时长是否超过 `8` 分钟
6. 如果超过 `8` 分钟，执行效率复盘并新增独立效率提升记录
7. 记录升级日志
8. 如有新开发规范，写入 [开发规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/开发规范.md)
9. 如有 AI 执行机制优化，写入 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)

补充规则：

- 效率复盘至少覆盖代码、架构、文档、流程、agent 协作、验证收尾六个维度
- 复盘结论必须区分“本轮已修正”和“后续持续执行”
- 如果复盘形成长期规则，还要同步更新 [效率提升.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/效率提升.md)
- 页面 / 交互开发时，应把高频轻量审查项前置为默认收尾检查，而不是等审查 agent 才首次发现
- 部署准备、升级 SQL、权限资源或静态快照类任务，在进入独立复核前，默认先完成一轮“无副作用预检”：
  - 目标父资源 / 菜单是否存在
  - 动态 ID、固定回退值和角色授权是否一致
  - 涉及“静态存储”语义时，`创建 / 更新 / 读取` 三条路径是否一起校验
  - 当前最小编译或验证命令是否已明确并实际执行

### 8.1 收尾结论最小集

每次正式开发收口时，至少要明确输出下面这组结论：

1. 是否存在范围漂移。
2. 是否存在回归风险。
3. 是否已经完成文档回填。
4. 当前结果是否允许进入下一阶段。

如果任一项结论仍不明确，本轮结果只能视为阶段性结果，不应标记为最终收口。

## 11. TraceID / 连续性资产增强规则（可选增强层）

为支持大型任务、多阶段交付和跨会话接力，项目可以按需启用 TraceID、`.project_state.json`、`.handoff/` 等连续性资产。

共享模板默认口径：

1. 正式文档入口链仍是首要权威来源。
2. TraceID 与状态记录用于补足续接、交接、回填和审计链路。
3. 如果项目规模较小、任务较短或团队没有这类需要，可以不把这套机制作为硬前置。

### 11.1 TraceID 定义
TraceID 是本任务在整个生命周期中的唯一标识符，通常格式为：`[IMPL-日期-模块-序号]`。
示例：`[IMPL-0401-MOD-01]`

### 11.2 推荐关联链
当项目启用该增强层时，建议将任务交付物尽量绑定到同一 TraceID：

1. **连续性状态**：`.project_state.json` 可记录 `active_task_id`。
2. **实施文档**：文件名或正文首行可包含此 ID。
3. **交接单**：`.handoff/[TraceID].md` 可作为跨会话摘要。
4. **Git Commit**：消息可带 `[TraceID]` 前缀。
   - 示例：`feat: [IMPL-0401-MOD-01] 增加核心表数据库迁移文件`
5. **升级日志**：变更主题可包含此 ID。

### 11.3 会话接力流程
若任务属于大型跨会话任务，且项目已启用这套连续性资产，AI 应执行以下动作：
1. 读取 `CLAUDE.md` 获取全局硬规则。
2. 再读取 `.project_state.json` 确定当前活跃 TraceID。
3. 再读取对应的 `.handoff/[TraceID].md`。
4. 验证当前代码状态是否与交接单一致。
5. 继续执行交接单中的 `Next Actions`。

### 11.4 自动化回填比对 (Diff-to-Doc)
在项目已启用 TraceID / 连续性资产，且任务进入收尾阶段（PHASE: VERIFICATION）时，建议执行：
1. 使用 `git diff HEAD` 提取本次 TraceID 下的全部改动。
2. 启动 `agency-technical-writer` 对比 Diff 与现有的 `后端.md / 前端.md`。
3. 自动识别新增接口、表字段或 UI 规范。
4. 自动产出文档补丁并完成回填。

一句话原则：
**文档入口优先；大型跨会话任务可按需启用 TraceID 与交接单。**
