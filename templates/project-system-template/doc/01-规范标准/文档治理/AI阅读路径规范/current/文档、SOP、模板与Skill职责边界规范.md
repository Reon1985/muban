# 文档、SOP、模板与 Skill 职责边界规范

## 1. 文档定位

本文件用于统一当前项目中以下载体的职责边界：

1. 长期规则文档
2. SOP
3. 模板
4. 版本包
5. 问题总结
6. 升级日志
7. Skill

本文件重点回答：

1. 同一条结论应该写进哪一层
2. 哪些内容只能写在正式文档里，不能只写在 skill 里
3. 哪些内容适合固化成模板或 SOP
4. 当规则、技能、版本交付物同时变化时，应如何分层回填

相关入口：

## 2. 总原则

当前项目统一采用一句话口径：

- 正式文档是权威源，skill 是路由器和执行提醒，不是平行规则库。

展开后默认遵循：

1. 长期稳定规则优先写进 `01-规范标准/`
2. 高频可重复动作优先写进 `02-标准操作手册/`
3. 可直接复用的填写骨架优先写成模板，仍放在 `02-标准操作手册/`
4. 某一版本的正式需求、实施、测试和收口材料写进 `03-版本包/`
5. 可复用的问题、排障和风险沉淀写进 `04-问题总结/`
6. 已发生的变化事实与证据写进 `05-升级日志/`
7. skill 只承接“如何让 AI 更稳定命中文档、路由和协作流程”，不承接长期权威规则正文

## 3. 载体职责矩阵

### 3.1 `01-规范标准`

回答的问题：

- 规则是什么
- 默认口径是什么
- 长期边界是什么

适合放：

1. 文档治理规则
2. AI 阅读路径规则
3. Agent / skill 协作边界
4. Hook、自动学习、去重治理等长期规则

不适合放：

1. 某一版的单次实施事实
2. 单次发版过程
3. 某个版本特有的测试清单

### 3.2 `02-标准操作手册`

回答的问题：

- 这类动作具体怎么执行

适合放：

1. 部署、发版、测试、回滚
2. 测试报告驱动问题收口操作卡片
3. AI 执行模板

不适合放：

1. 长期规则原文
2. 某个版本的具体业务方案
3. 单次问题修复事实

### 3.3 模板

模板属于 `02-标准操作手册/` 的可复用骨架。

回答的问题：

- 这类文档应该按什么骨架填写

适合放：

1. 复杂任务蓝图模板
2. 新项目接入模板
3. 测试报告问题批量收口实施模板

不适合放：

1. 解释长期规则边界
2. 替代正式规则正文
3. 承担某个版本的最终正式结论

### 3.4 `03-版本包`

回答的问题：

- 这一版具体要交付什么
- 这一版具体怎么落地

适合放：

1. 当前版本需求
2. 页面规则和 UI 方案
3. 数据库、API、实施文档
4. 测试用例、测试报告、问题修复清单

关键原则：

1. 同一版本内持续修复，继续在原版本包内演进
2. 修 Bug 不默认新开版本
3. 只有当修复已经扩成新的独立业务范围或新的正式交付边界时，才考虑新开版本包

### 3.5 `04-问题总结`

回答的问题：

- 这个问题下次再遇到时如何快速复用

适合放：

1. 排障分析
2. 已知限制
3. 风险清单
4. 版本特有问题总结

关键原则：

1. 通用可复用的问题优先放 `current/`
2. 明显属于某个版本特有的问题再进 `versions/<版本>/`

### 3.6 `05-升级日志`

回答的问题：

- 已经发生了什么变化
- 有哪些证据和结论

适合放：

1. AI 规则升级日志
2. 单次发版与回归日志
3. 效率提升日志
4. 文档体系调整日志

关键原则：

1. 升级日志只做证据层，不做长期规则主承接层
2. 如果规则本身变化，规则正文必须回写到 `01-规范标准/` 或 `02-标准操作手册/`

### 3.7 Skill

回答的问题：

- AI 进入这个任务时，应先读什么
- 这类任务该如何路由
- 哪些边界不能越过

适合放：

1. 启动阅读顺序
2. 任务分型规则
3. 文档优先、实施门禁、复核要求等执行提醒
4. 仓库 adapter 的路径与例外说明

不适合放：

1. 单独维护一套长期规则正文
2. 和正式文档平行维护同一条完整规则
3. 只在 skill 中出现、正式文档完全没有落点的长期治理规则

## 4. 权威源优先顺序

当同一条结论在多个载体同时出现时，默认按下面顺序判断权威源：

1. `01-规范标准/` 的正式规则正文
2. `02-标准操作手册/` 的 SOP 正文
3. `03-版本包/` 中当前版本的正式正文
4. `04-问题总结/` 的可复用问题沉淀
5. `05-升级日志/` 的事实与证据
6. skill 中的执行提醒与路由说明

一句话：

- skill 可以引用规则，但不应反向取代正式规则。

---

## 5. 默认判断顺序

每次新增 AI 规则、文档、模板、skill 或修复结论时，默认按下面顺序判断落点：

1. 这是长期规则，还是单次事实
2. 这是“怎么判断”，还是“怎么执行”
3. 这是可复用骨架，还是本轮正式结论
4. 这是当前版本交付物，还是跨版本通用知识
5. 这是排障经验，还是交付方案
6. 这条内容是否必须让 AI 启动时优先命中

对应落点：

1. 长期规则 -> `01-规范标准/`
2. 执行动作 -> `02-标准操作手册/`
3. 可套用骨架 -> `02-标准操作手册/AI执行模板/`
4. 当前版本交付 -> `03-版本包/<版本>/`
5. 问题与排障 -> `04-问题总结/`
6. 变化证据 -> `05-升级日志/`
7. AI 路由提醒 -> skill

## 5.1 常见任务决策表

下面这张表用于快速判断高频场景该先落在哪一层：

| 场景 | 主落点 | 辅助落点 | 默认动作 |
| --- | --- | --- | --- |
| 新需求 / 新版本增量 | `03-版本包/<版本>/` | `05-升级日志/` | 先进入目标版本包，按需求、页面、规则、实施文档推进 |
| 当前版本内的单点 Bug 修复 | `03-版本包/<当前版本>/` | `05-升级日志/` | 不默认新开版本，优先在当前版本包内收口 |
| 测试报告驱动的一批 Bug 修复 | `03-版本包/<当前版本>/` | `02-标准操作手册/`、`05-升级日志/` | 先冻结问题清单，再集中修复、统一发版、统一验证 |
| 排障分析 / 可复用问题经验 | `04-问题总结/` | `05-升级日志/` | 如果下次还可能复用，优先沉淀到问题总结 |
| 发版、联调、回归事实与证据 | `05-升级日志/` | `03-版本包/<版本>/` | 记录发生了什么、验证了什么，不替代正式方案正文 |
| 可重复执行的固定流程 | `02-标准操作手册/` | `02-标准操作手册/AI执行模板/` | 把“怎么执行”固化成 SOP；如需要固定骨架，再补模板 |
| AI 路由、启动顺序、执行提醒 | skill | `01-规范标准/`、`02-标准操作手册/` | 先确认正式文档已有权威规则，再把最小提示写入 skill |

补充判断：

1. 如果同一件事同时触发多个层级，先确定主落点，再决定是否补辅助落点
2. 如果只是当前版本收口，不因为“看起来像升级”就新开版本包
3. 如果结论未来仍会被反复执行或判断，优先补进 `01` 或 `02`，不要只停留在 `05`

## 6. 文档与 Skill 的同步规则

当 AI、规则、skill 行为调整时，默认至少做以下同步：

1. 先更新正式文档
2. 再更新对应 skill 的路由和提醒
3. 再更新目录 `README.md`
4. 如属于治理规则升级，再更新 [AI能力与规则升级日志.md](__PROJECT_ROOT__/doc/05-升级日志/AI执行体系/AI能力与规则升级日志.md)

强约束：

1. 不允许只改 skill，不改正式文档，就把长期规则视为生效
2. 不允许把 `README` 当作唯一规则正文
3. 不允许把升级日志当作规则正文

## 7. 常见错误

### 7.1 错误：把 skill 当正式规则库

问题：

1. 规则只在 skill 中存在，不利于追溯
2. 不同 skill 各写一套，容易冲突

正确做法：

1. 正式规则先写进文档
2. skill 只保留路由、触发、边界和最小提示

### 7.2 错误：把日志当规则正文

问题：

1. 能看到变化，但看不到当前正式口径
2. 后续 AI 容易从历史日志读出过期结论

正确做法：

1. 规则正文回写到 `01` 或 `02`
2. 日志只保留“这次改了什么”

### 7.3 错误：把模板写成完整规则正文

问题：

1. 模板越来越长，难以直接套用
2. 规则与骨架混杂，维护成本高

正确做法：

1. 模板只保留必填骨架和关键门禁
2. 详细规则通过引用指向正式规则正文

### 7.4 错误：当前版本修复一来就新开版本

问题：

1. 版本包碎片化
2. 当前版本收口事实被拆散

正确做法：

1. 当前版本修复优先在当前版本包内收口
2. 只有形成新的正式升级范围时，才新开版本包

---

## 8. 完成标准

当前体系被视为边界清晰，至少满足：

1. 能回答每条内容该写进哪一层
2. 正式规则不只存在于 skill 中
3. skill 不再与正式文档平行维护同一条完整规则
4. README、正文、模板、日志之间不存在长期职责错位
5. 规则升级时，能同时回写正式文档、skill 和升级日志
