# 飞书 Bug 修复与开发确认提示词模板

## 1. 文档定位

本模板是给“已接入飞书问题表”的项目预留的专项执行骨架。

它用于处理两类待办：

1. `待修复` 类记录的修复与回写
2. `待开发` 类记录的范围判断、版本归属和开发确认回写

本模板默认保持平台无关、项目无关，不内置任何真实 `Base Token`、`Table ID`、`View ID`、账号或环境地址。

## 2. 使用前提

使用前先补齐以下项目私有信息：

1. 飞书多维表格地址
2. `Base Token`
3. `Table ID`
4. `View ID`
5. 当前项目允许回写的字段清单
6. 当前项目的版本包入口和问题修复归档规则

如果项目尚未接入飞书问题表，可跳过本模板。

## 4. 变量清单

在真正执行前，把下面占位符替换成项目私有值：

- `<BASE_TOKEN>`
- `<TABLE_ID>`
- `<VIEW_ID>`
- `<RECORD_ID>`
- `<CURRENT_VERSION>`
- `<RESULT>`

## 5. 推荐命令骨架

### 5.1 读取字段结构

```bash
lark-cli base +field-list \
  --base-token '<BASE_TOKEN>' \
  --table-id '<TABLE_ID>'
```

### 5.2 读取当前视图记录

```bash
lark-cli base +record-list \
  --base-token '<BASE_TOKEN>' \
  --table-id '<TABLE_ID>' \
  --view-id '<VIEW_ID>' \
  --limit 500
```

### 5.3 回写单条记录

```bash
lark-cli api PUT '/open-apis/bitable/v1/apps/<BASE_TOKEN>/tables/<TABLE_ID>/records/<RECORD_ID>' \
  --data '{"fields":{"修复结果":"<RESULT>","测试结果":"待发版"}}'
```

## 6. 直接可用提示词骨架

```markdown
你是当前项目的问题修复执行代理。你的任务不是只读问题表，而是要把问题记录、当前正式需求、当前版本包和本地文档回填一起收口。

先做上下文读取，再执行问题记录处理。默认阅读顺序：

1. `CLAUDE.md`
2. `doc/README.md`
3. `doc/00-入口/AI执行入口.md`
4. `doc/00-入口/当前生效版本说明.md`
5. `doc/01-规范标准/文档治理/AI阅读路径规范/current/问题修复文档归档与执行规范.md`
6. `doc/01-规范标准/文档治理/AI阅读路径规范/current/测试报告驱动问题批量收口规则.md`
7. `doc/02-标准操作手册/AI执行模板/README.md`
8. 当前主线版本包和相关需求文档

然后读取问题表：

- Base Token: `<BASE_TOKEN>`
- Table ID: `<TABLE_ID>`
- View ID: `<VIEW_ID>`

你的处理范围只包含下面两类记录：

1. `测试结果 = 待修复` 且 `状态` 不为空
2. `测试结果 = 待开发` 且 `状态` 不为空

执行要求：

1. 先把问题映射到当前正式版本和模块
2. 对照当前正式需求、页面规则、已有实施文档和测试清单判断是否在已确认范围内
3. 如需代码修改但尚无正式实施文档，先补齐问题修复实施文档
4. 完成后同步回填本地文档，不允许只回写表格不回填仓库
5. 对 `待修复` 记录回写 `修复结果` 和 `测试结果 = 待发版`
6. 对 `待开发` 记录回写判断结论，并同步推进到 `开发确认` 或项目约定状态
```

## 8. 模板边界

1. 本文件只提供专项提示词骨架，不保存真实凭据。
2. 真实表结构、字段口径和回写策略必须由项目落地时补齐。
3. 如果项目改用其他问题跟踪平台，可复制本文件并改造成对应平台模板。
