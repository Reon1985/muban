## 2. 使用原则

1. 本目录承接“可以直接套用”的 AI 执行模板。
2. 长期原则写在 `01-规范标准/`，模板和执行骨架写在 `02-标准操作手册/`。
3. 使用模板时可以裁剪，但不应删除关键门禁和回填项。
4. 如果任务来源于测试报告驱动的问题修复，优先使用“测试报告问题批量收口实施模板”，不要直接从空白实施文档起草。
5. 如果任务属于单点 Bug、线上反馈或排障发现的通用问题修复，优先使用“问题修复实施模板”。
6. 如果项目后续接入飞书或其他缺陷跟踪平台，可在当前目录继续扩展专项执行提示词模板。

## 3. 模板选择

| 场景 | 优先模板 | 说明 |
| --- | --- | --- |
| 测试报告、联调报告、QA 缺陷清单驱动修复 | [测试报告问题批量收口实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/测试报告问题批量收口实施模板.md) | 适合冻结问题清单、集中修复、统一发版、统一验证 |
| 单点 Bug、线上反馈、排障发现的通用修复 | [问题修复实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/问题修复实施模板.md) | 适合非测试报告驱动的单次修复与归档判断 |
| 来源于飞书 Bug 跟踪表，且要同时处理 `待修复 -> 待发版` 与 `待开发 -> 开发确认` 回写 | [飞书Bug修复与开发确认提示词模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/飞书Bug修复与开发确认提示词模板.md) | 适合只查阅 `状态` 非空记录；对 Bug 回写 `修复结果` 与 `测试结果`，对新需求除回写 `修复结果` 与 `测试结果` 外，还同步回写 `状态 = 待开发确认` |
| 不确定属于哪类问题修复 | 先看 [问题修复文档归档与执行规范.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/问题修复文档归档与执行规范.md) | 先判定归档层和来源，再选模板 |
