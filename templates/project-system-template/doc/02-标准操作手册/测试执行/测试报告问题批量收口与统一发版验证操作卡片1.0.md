# 测试报告问题批量收口与统一发版验证操作卡片 1.0

## 1. 用途

本卡片用于指导“测试报告驱动的问题修复”如何一次性收口、统一发版和统一验证。

适用场景：

1. 问题来源是测试报告、联调报告或 QA 缺陷清单
2. 本轮修复完成后需要发版到服务器验证
3. 希望尽量减少发版次数

不适用场景：

1. 安全漏洞紧急止血
2. 生产阻断或数据错误持续扩大
3. 明确的高优先级线上故障

如不适用，必须在实施文档中写明豁免原因。

---

## 2. 先做适用性判断

先判断本轮是否属于“测试报告驱动修复”：

1. 如果是，必须按本卡片执行
2. 如果不是，但仍然需要发版验证，可直接走常规发版 SOP
3. 如果属于紧急例外，必须在实施文档中写清“豁免原因”

对应长期规则：

- [测试报告驱动问题批量收口规则.md](__PROJECT_ROOT__/doc/01-规范标准/文档治理/AI阅读路径规范/current/测试报告驱动问题批量收口规则.md)

## 3. 标准执行步骤

### 步骤 1：建立实施文档并冻结问题清单

优先使用：

- [测试报告问题批量收口实施模板.md](__PROJECT_ROOT__/doc/02-标准操作手册/AI执行模板/测试报告问题批量收口实施模板.md)

实施文档中至少补齐：

1. 来源测试报告
2. 本轮问题清单
3. 冻结问题清单
4. 统一发版验证计划
5. 如不适用本规则，豁免原因

### 步骤 2：本地集中修复与本地自测

执行要求：

1. 先集中修复本轮必修复项
2. 本地完成最小回归验证
3. 中途不反复发版到服务器

### 步骤 3：发版前跑门禁检查

在正式发版前，先执行：

```bash
bash .claude/scripts/ai-exec-hooks/pre-release-batch-gate.sh
```

如果返回阻断信息，先补齐实施文档，再继续发版。

补充说明：

1. 只写章节标题、不填写实际内容，不算通过门禁
2. 如走例外豁免，必须填写真实原因，不能只保留“豁免原因”空字段

### 步骤 4：统一发版

后端发版参考：

- [目标环境后端发版操作卡片1.0.md](__PROJECT_ROOT__/doc/02-标准操作手册/部署与发版/目标环境后端发版操作卡片1.0.md)

前端发版参考：

- [目标环境前端静态发布操作卡片1.0.md](__PROJECT_ROOT__/doc/02-标准操作手册/部署与发版/目标环境前端静态发布操作卡片1.0.md)

### 步骤 5：统一验证

发版后按测试报告整轮验证，不允许只抽测一个页面或一个接口就结束。

至少验证：

1. 本轮必修复项
2. 关联回归项
3. 发版影响到的关键业务链路

### 步骤 6：回填收口

本轮验证结束后，至少回填：

1. 实施文档结论
2. 测试报告结论
3. 升级日志
4. 必要的问题总结

---

## 4. 发版前最小检查清单

正式发版前，至少确认：

1. 本轮问题清单已冻结
2. 本轮必修复项已完成代码修复
3. 本地已完成最小自测
4. 实施文档已写明统一发版验证计划
5. 前后端、数据库、验证路径都已准备完成

---

## 5. 常见错误

### 5.1 错误：边修边发版

问题：

1. 发版次数过多
2. 验证结论被前后版本污染
3. 很难判断到底哪次修复真正生效

正确做法：

1. 先冻结本轮问题清单
2. 集中修复
3. 再统一发版

### 5.2 错误：发版后只抽测几个点

问题：

1. 无法证明测试报告问题整体关闭
2. 关联回归容易漏掉

正确做法：

1. 发版后按测试报告逐条验证
2. 同时补最小回归

### 5.3 错误：例外豁免不留痕

问题：

1. 下次无法追溯为什么中途单独发版
2. 门禁无法区分“合理例外”和“随意绕过”

正确做法：

1. 在实施文档中明确写出豁免原因

---

## 6. 一句话顺序

```text
测试报告
→ 冻结问题清单
→ 集中修复与本地自测
→ 发版前门禁检查
→ 统一发版
→ 整轮统一验证
→ 回填收口
```
